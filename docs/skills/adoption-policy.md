# 外部 Skills 採用ポリシー

## 基本原則

**「便利そうだから入れる」は禁止。**

外部Git由来の Skills / rules / スクリプト等は、**セキュリティ確認と PM 承認を前提に利用可能**とする。
ただし、中身を確認せずに導入することは許可しない。人気やスター数だけで採用判断しない。

### 方針

- 利用可否は**セキュリティ・保守性・自社運用との整合**で判断する
- **直接導入よりも、自社向けに再構成することを基本**とする
- 検証なしの本番利用は禁止
- 初回は必ずサンドボックスまたはローカル検証環境で確認する
- **情報漏洩リスクが少しでも疑われる場合は採用しない**
- 問題がある場合は不採用とする。判断に迷う場合も不採用を基本とする

## 導入区分

外部Git由来の資産は、以下の区分で管理する。

| 区分 | 内容 | PM承認 | 承認台帳への登録 |
|------|------|--------|-----------------|
| 参考利用 | 考え方だけ取り込む。コード自体は使わない | 不要 | 不要 |
| 部分採用 | 一部の記述・観点を自社 skills/rules に組み込む | 必要 | 推奨 |
| 限定導入 | 特定案件または検証環境のみで使う | 必要 | 必要 |
| 正式採用 | 承認済み資産として継続利用する | 必要 | 必要 |

### 導入の優先順位

1. **内容を確認して自社 skills として再構成する**（最も推奨）
2. 問題ないと判断したものを限定採用する
3. 継続利用する場合のみ承認済み資産として正式登録する

### 原則は「参考利用」または「部分採用」

- 可能な限り自社管理下に取り込み、内容を固定して使う
- 外部参照のまま継続依存する場合は、**その理由を承認台帳に記録する**
- 外部リポジトリの更新に自動追従しない。バージョンを固定する

## 採用判断基準

外部Git由来資産の採用を検討する場合、以下の基準をすべて満たすこと。

### 必須基準（すべて満たすこと）

| # | 基準 | 確認方法 |
|---|------|---------|
| 1 | 明確な必要性がある | 内部 Skills で代替できないか検討済み |
| 2 | ソースコードが公開されている | リポジトリを確認 |
| 3 | コードの内容を確認した | 目視レビュー実施済み |
| 4 | 不審な依存関係がない | package.json / pnpm-lock.yaml 等を確認 |
| 5 | 意図しない外部通信がない | コードを確認 |
| 6 | 情報漏洩リスクがない | 情報漏洩チェックを実施済み |
| 7 | ライセンスが商用利用に適合する | LICENSE ファイルを確認 |
| 8 | 特定の commit hash または tag で固定している | main / master 等の可変参照のまま導入しない |
| 9 | サンドボックスで検証済み | ローカル検証環境で動作確認済み |
| 10 | PM が採用を承認した | 承認記録あり |

### 即時不採用の基準

以下のいずれかに該当する場合は **採用しない**。

- **業務上不要な**外部送信処理がある（telemetry・analytics 含む）
- 環境変数・認証情報・ローカルファイルの取得範囲が広すぎる
- 実行内容が明確に説明できない
- 難読化・圧縮されたコードが含まれる
- `npx` やスクリプト実行の安全性が確認できない
- 依存関係が追い切れない
- 自社の情報取扱ルールや AI 利用ルールと矛盾する

> **外部通信が業務上必要な場合**: 通信の目的・送信内容・停止可否を確認し、PM承認を必須とする。通信先・内容を承認台帳に記録する。

### 推奨基準（可能な限り満たすこと）

| # | 基準 | 確認方法 |
|---|------|---------|
| 1 | メンテナンスされている | 直近6ヶ月以内のコミットがあるか |
| 2 | 他プロジェクトでの利用実績がある | ダウンロード数、Issue、利用報告 |
| 3 | テストが書かれている | テストコードの存在を確認 |
| 4 | ドキュメントが整備されている | README、使い方の記載 |

> **注意**: 利用実績、ダウンロード数、スター数は参考情報であり、安全性を保証するものではない。推奨基準をすべて満たしていても、必須基準の確認は省略できない。

## 情報漏洩チェック手順

外部Git由来資産を採用候補にする前に、**情報漏洩リスクを最優先で確認する**。
詳細なチェックリストは `docs/skills/external-skills-evaluation-checklist.md` を参照。

### 確認の観点

1. **外部通信**: HTTP/HTTPS送信、Webhook、telemetry、analytics がないか
2. **情報抜き取り**: 環境変数、APIキー、SSH鍵、`.env`、認証ファイル等を収集していないか
3. **危険な実行**: `curl|bash`、難読化コード、不明なダウンロード処理がないか
4. **権限の過剰さ**: 本来不要なファイルアクセスやディレクトリ走査がないか
5. **依存関係**: 依存先に危険な処理が潜んでいないか

### 確認の姿勢

- 「怪しくないことを確認する」姿勢で臨む
- 少しでも不明点があれば不採用に倒す
- 「便利そう」「人気がある」は判断理由にしない

## 依存関係の確認手順

1. `package.json` / `pnpm-lock.yaml` 等の依存一覧を確認する
2. 依存パッケージの数が多すぎないか確認する
3. 各依存パッケージが信頼できるものか確認する
4. `pnpm audit` で既知の脆弱性がないか確認する（ただし未知の攻撃は検出できない）
5. `pnpm why パッケージ名` で推移的依存の経路を確認する（直接使っていなくても混入するリスクがある）
6. 依存先の `postinstall` / `preinstall` / `prepare` スクリプトを確認する
7. 最近作られたパッケージや名前が紛らわしいパッケージに注意する（typosquatting）
8. バージョンは `^` ではなく固定を推奨する（`pnpm add --save-exact`）
9. lockfile の変更がある場合は差分をレビューする

## npx / 外部スクリプト実行の扱い

### 原則

- `npx` で実行するパッケージは、事前にパッケージ名と内容を必ず確認する
- 初めて実行するパッケージは PM に報告してから実行する
- `npx` の `--yes` フラグは安易に使わない

### 確認項目

| # | 確認項目 | 内容 |
|---|---------|------|
| 1 | パッケージ名 | 正しい名前か（typosquatting でないか） |
| 2 | 公開元 | npm レジストリでパッケージ情報を確認 |
| 3 | ダウンロード数 | 極端に少ない場合は注意 |
| 4 | 最終更新日 | 長期間更新がない場合は注意 |
| 5 | 実行内容 | 何をするパッケージか理解しているか |
| 6 | 権限 | ファイル操作やネットワーク通信の範囲 |

### 外部スクリプトの実行

- 出所不明のスクリプトを `curl | bash` 等で実行しない
- 実行前にスクリプトの内容をダウンロードして目視確認する
- ビルドスクリプトやインストールスクリプトの中身も確認する

## PM 承認ルール

### 承認が必要な場面

- 外部Git由来資産を部分採用・限定導入・正式採用するとき
- 外部 Skills のバージョンを更新するとき
- 新しい `npx` パッケージを初めて実行するとき
- 外部スクリプトを実行するとき

### 承認フロー

```
1. 担当者が採用を提案
   └→ 採用理由、代替手段の検討結果、導入区分を記載

2. 情報漏洩チェックを実施
   └→ external-skills-evaluation-checklist.md に沿って確認

3. サンドボックス検証を実施
   └→ ローカル検証環境で動作確認

4. PM に承認申請
   └→ 提案内容 + 情報漏洩チェック結果 + 検証結果を提出

5. PM が承認 / 却下
   └→ 承認の場合、approved-skills-register.md に記録

6. 導入
   └→ 承認された導入区分に従って導入
   └→ 特定の commit hash / tag を固定して導入（可変参照で本番導入しない）
   └→ approved-skills-register.md に commit hash / tag を記録
```

## サンドボックス検証ルール

外部Git由来資産を本番プロジェクトに導入する前に、**必ずサンドボックス環境で検証**すること。
検証なしの本番利用は禁止。

### 検証項目

- [ ] 期待通りの動作をするか
- [ ] エラー時の挙動は安全か
- [ ] 不要なファイルの作成・変更がないか
- [ ] 意図しない外部通信がないか（ネットワーク監視で確認）
- [ ] 環境変数やローカルファイルへの不正アクセスがないか
- [ ] 既存の Skills やルールと競合しないか
- [ ] プロジェクトの設定に悪影響がないか
- [ ] 自社の情報取扱ルールと矛盾しないか

### 検証環境のルール

- **検証には本番認証情報を使わない**（APIキー、トークン、パスワード等）
- ダミーデータまたはマスキングデータを使う
- 可能ならネットワーク制限された環境で検証する
- 検証後、検証環境に残った認証情報やデータを削除する

### 検証方法

1. 別ディレクトリまたはブランチでテスト環境を用意する
2. 対象資産を導入し、基本的な操作を一通り試す
3. 上記の検証項目をチェックする
4. 問題がなければ PM に検証結果を報告する
5. 問題がある場合は不採用とする

## 再評価トリガー

正式採用済みの外部Git由来資産は、以下のタイミングで再評価を行う。

| トリガー | 対応 |
|----------|------|
| バージョン更新時 | 更新内容を確認し、再度セキュリティチェックを実施 |
| 依存関係変更時 | 新規・更新された依存パッケージを確認 |
| 脆弱性報告時 | 即時に影響範囲を確認し、継続可否を判断 |
| 半年ごとの棚卸し | 利用状況・メンテナンス状況・代替手段の有無を確認 |

## 不採用・削除・使用停止の基準

### 不採用の基準

以下に該当する場合は採用しない。

- セキュリティ上の脆弱性が報告された
- 情報漏洩リスクが疑われる事象が発覚した
- メンテナンスが6ヶ月以上停止している
- 内部 Skills で十分に代替できるようになった
- 実際の運用で使われていない（3ヶ月以上未使用）
- 依存関係に問題が発見された
- 自社の情報取扱ルールや AI 利用ルールと矛盾が生じた

### 問題発覚時の対応

1. **まず新規採用と更新を停止する**
2. 既存利用中のものは影響範囲を確認する
3. 継続可否を再評価し、PM が最終判断する
4. 不採用・削除の場合は、承認台帳のステータスを更新する
