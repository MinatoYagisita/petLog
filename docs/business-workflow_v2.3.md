# AI開発代行サービス 業務フロー設計書

## 体制

| 役割 | 担当 | 主な責務 |
|------|------|----------|
| PM | 1名 | 要件定義主導・仕様承認・レビュー・最終品質判断・重要な顧客判断 |
| 開発メンバー（若手） | 2名（+4月〜1名） | 担当案件の実務オーナー。開発・テスト・納品準備・顧客との通常連絡を主体的に推進 |

### 役割分担の基本思想

**「若手が自分の案件を自分で回す。PMはゲートで止める。」**

- 若手は単なる作業者ではなく、**自分の担当案件の実務オーナー**である
- 若手にとっては「練習」ではなく「自分の案件」。自分で考えて進め、必要なところだけPMにレビューを求める
- PMは案件のすべてを細かく面倒見る役割ではない
- PMは**最終責任者・レビュー担当・要所での判断者**に寄せる
- PMが過剰に伴走しすぎる運用は避ける
- **「PMが抱える」のではなく、「若手が進め、PMがゲートで止める」**構造にする

### PMの役割

| 領域 | PMがやること | PMがやらないこと |
|------|------------|----------------|
| 要件定義・仕様 | 初回ヒアリング主導、要件定義・仕様策定の主導、仕様承認 | 議事録作成、資料整理（若手が担う） |
| 開発 | レビュー依頼への対応、仕様解釈の最終判断 | 日々の進捗管理の細かい追いかけ、随時の伴走 |
| テスト | テスト仕様書のレビュー・抜け漏れ確認、品質の最終判断 | テスト実施、証跡整理 |
| 納品・検収 | 仕様解釈の争い、追加要望判定、重大不具合判断、検収条件の最終判断 | 納品準備、受入テスト指摘の取りまとめ、軽微な顧客やりとり |
| 顧客対応 | 仕様判断・契約判断・追加費用判断・トラブル時の対応 | 日常的な連絡・日程調整・確認事項の整理 |
| 保守 | エスカレーション判断・追加開発判断・契約範囲判断 | 一次対応、対応ログの記録 |

### 若手の役割

- 自分の担当案件の実務オーナーとして、開発・テスト・納品準備を主体的に進める
- 顧客への一次連絡・日程調整・確認事項整理など、判断を伴わない範囲の顧客対応を行う
- 担当案件について、進捗・課題・不明点を自分から整理してPMに上げる
- PMに「言われたことをやる」のではなく、自分で考えて進める
- 判断に迷うとき・仕様解釈が曖昧なとき・揉めそうなときだけPMに相談する
- 議事録作成・資料整理・論点整理はヒアリング段階から若手が主担当で行う

### 顧客窓口の考え方

**「通常連絡は若手、重要判断はPM」**

- 対顧客の**最終責任者**はPM
- ただし、日常的な顧客連絡のすべてをPMが持つわけではない
- 案件進行に必要な通常連絡（日程調整、確認依頼、納品物の送付等）は若手が対応する
- 仕様判断・契約判断・追加費用判断・トラブル時のみPMが前に出る

### 事業方針・利益設計

#### 基本方針

立ち上げ初期は、最初から高利益率を追わない。
まずは**低固定費で黒字化すること**を優先する。

- 若手メンバーはSESで出すと月30万〜35万円程度の水準
- したがって、まずは**1人あたり月40万円の売上が立てば十分に良い**という考え方を採用する
- ただし、これは「案件単価を無条件に安くする」という意味ではない
- **若手の開発工数部分の社内基準単価**として月40万円を置く考え方である
- PM工数・顧客折衝・レビュー工数まで全部込みで40万円と考えないこと

#### 価格改定の方針

- 単価を下げるのは、**テンプレート化・フォーマット化・AI活用によって本当に高回転で回せると実測で確認できた後**とする
- 「AIで速く作れるから安くする」ではなく、まずは**速く・多く回して、実測で余裕が見えてから商品別に価格調整する**
- 同じ単価帯でより多く・より速く回せる状態を作ることが先
- 「回せそうだから先に安く売る」は禁止。**実績を見てテンプレ案件だけを値下げ/定額化する**方針にする

#### 採算管理の考え方

立ち上げ初期は、1案件ごとの最大利益よりも、**赤字案件を踏まないこと**を優先する。

採算管理上は以下を切り分けて管理する：
- 若手開発工数（社内基準：40万円/人月）
- PM工数（別枠で評価）
- 営業インセンティブ（売上の2%）
- PMインセンティブ（売上の3%）
- AIツール費・SaaS利用費
- その他管理費（通信費、端末費等）

#### インセンティブ設計

| 対象 | 料率 | 考え方 |
|------|------|--------|
| 営業 | 売上の2% | 案件獲得に対する成果報酬 |
| PM側メンバー（3名想定） | 売上の3% | 最終責任・仕様判断・品質判断・レビュー責任を担うことへの配分。日常実務を全部やることへの報酬ではない |

- エンジニア（若手）には作った分に応じたインセンティブは設定しない
- 支給タイミングは**入金確認後**
- PMインセンティブは、若手の自走を前提としたうえで、PMが要所で品質と方向性を担保することへの対価
- 案件粗利が著しく低い場合は、インセンティブを調整する余地を残す（運用判断）

#### 案件分類の考え方

案件は以下の2種類に分類し、価格設計・リソース計画を変える。

**テンプレ型案件：**

| 条件 | 内容 |
|------|------|
| 規模 | 小規模（画面5以下） |
| 仕様 | 明確。ヒアリング1回で確定できる |
| 外部連携 | なし、または極小 |
| 権限 | 単純（なし or 管理者/一般の2ロール程度） |
| 顧客確認 | 速い（翌営業日以内に返答がある） |
| テンプレ適合 | 既存テンプレートに乗る |

扱い：
- 高回転を前提にできる
- 若手が月2〜3件回せる可能性がある
- 将来的に定額化・低価格化の候補になる
- **ただし、定額化するのは実績で回転率が確認できた後**

**個別対応案件：**

| 条件 | 内容 |
|------|------|
| 要件 | 曖昧。ヒアリング複数回が必要 |
| 外部連携 | あり |
| 権限 | 複雑（3ロール以上、画面単位の制御等） |
| 運用条件 | 重い（データ移行、既存システムからの切り替え等） |
| PM判断 | 多い。仕様の判断が頻繁に発生する |
| 顧客確認 | 遅い、または決裁者が不明確 |

扱い：
- テンプレ案件と同じ価格思想にしない
- PM工数・調整工数をしっかり乗せる
- 安易に値下げしない
- 高回転前提にしない

### 品質を支える共通基盤（重要）

若手が開発を担当するが、品質は個人のスキルに依存させない。
以下の**事前に用意した共通基盤**を全案件で使用することで、一定の品質水準を担保する。

品質共通基盤は「ファイルを置くだけ」ではなく、**運用設計された基盤**として扱う。

#### 全体設計方針

- `CLAUDE.md` に全部詰め込まない。**短く保ち、詳細は分割する**
- Skills は「常時読ませるルール」ではなく、**必要なフェーズ・用途でだけ読むオンデマンド手順書**として設計する
- 外部Skills は人気だけで採用しない。**PM承認・中身確認・依存関係確認・サンドボックス検証**を経て採用する
- Hooks / Orchestration は将来の拡張候補。初期段階では導入しない

#### 28. CLAUDE.md（AI指示ルール・コード規約）

**役割：** プロジェクト全体の最低限の共通ルールを置く。AIツールが最初に読む「基本方針書」。

**設計方針：**
- `CLAUDE.md` は全体方針・最低限の共通ルールだけを置く（肥大化させない）
- 詳細ルールは `.claude/rules/` に分割する
- CLAUDE.md が長くなりすぎると守られにくくなるため、分割可能な構成にする

**ファイル構成：**

```
CLAUDE.md                          ← 全体方針・最低限のルール
.claude/rules/
  ├── code-style.md               ← コーディング規約・命名規則
  ├── testing.md                  ← テスト基本方針
  ├── security.md                 ← セキュリティ基本方針
  └── ui.md                       ← UI/UX基本方針
```

**CLAUDE.md に書く内容（例）：**
- プロジェクトの概要・目的
- 使用する技術スタック
- コーディングの基本原則（簡潔・可読性重視・仕様書に忠実）
- 「詳細は .claude/rules/ を参照」の誘導
- お客様データをAIに入れないルールの再確認

**現段階の運用：** まずは CLAUDE.md + 数本の rules ファイルで始める。案件を重ねながら分割の粒度を調整する。

#### 29. skills：テスト（モジュール化されたテスト基盤）

**役割：** テスト工程の品質を、若手でも一定水準で回せるようにする手順書群。

**設計方針：**
- 1つの大きな「標準テストパターン一式」ではなく、**用途別に分けたSkillsモジュール群**として設計する
- 常時読み込むのではなく、必要なフェーズ・用途でだけ使う
- 若手でも同じ観点でテストできるよう、手順書として機能する構造にする
- PMがテスト観点を補強するときにも使えるようにする

**モジュール構成：**

| モジュール | 用途 | 使用タイミング |
|-----------|------|--------------|
| `test-spec-creator` | 仕様書からテスト観点・テスト項目を起こす | Phase 5 テスト仕様書作成時 |
| `boundary-and-validation-test` | 境界値、未入力、型不正、重複、最大値/最小値のテスト観点 | Phase 5 テスト仕様書作成時 |
| `test-executor` | テスト実施手順・証跡の残し方・OK/NG判定基準 | Phase 5 テスト実施時 |
| `acceptance-test-support` | 受入テスト観点、指摘票の書き方、P1/P2/P3分類基準 | Phase 6 検収対応時 |
| `bug-repro-template` | 不具合の再現手順の標準フォーマット | Phase 5〜6 不具合発生時 |

#### 30. skills：デザイン（UI品質・UX・アクセシビリティ基盤）

**役割：** 業務ツールとして破綻しないUI/UXを安定化させる。「美しさ」よりも「使いやすさと一貫性」を重視する。

**対象範囲：**
- レイアウト・余白のルール
- 配色・コントラスト
- タイポグラフィ（フォント・サイズ・行間）
- コンポーネント利用ルール（ボタン・フォーム・テーブル等の統一）
- 一覧画面・検索UIのパターン
- フォームUIのパターン（入力・バリデーション表示・エラー表示）
- 操作の一貫性（同じ操作が同じ結果を生む）
- アクセシビリティの最低ライン（コントラスト比、フォーカス順序、altテキスト等）

**設計方針：**
- 外部Skillsをそのまま使わない。**中身を確認したうえで、自社のdesign skillsに落とし込む**
- 案件ごとのデザインばらつきを抑えるだけでなく、**最低限の使いやすさを安定化させること**が目的
- 「デザインの美しさ」ではなく「業務ツールとして破綻しないUI/UX」を重視する

#### 31. skills：セキュリティチェック（アプリ＋開発基盤の安全確認）

**役割：** アプリケーションの脆弱性チェックに加え、**AI開発基盤への外部依存リスク**も確認対象とする。

**対象範囲：**

**アプリケーションのセキュリティ（従来）：**
- 入力値検証
- 認証・認可
- SQLインジェクション対策
- XSS対策
- CSRF対策
- セッション管理
- エラー情報の漏洩防止

**開発基盤のセキュリティ（追加）：**
- 外部Skills採用時の安全確認

**外部Skills採用時の審査ルール：**

外部Skillsを導入する場合は、以下の審査を必ず実施する。「便利そうだから入れる」は禁止。

| 審査項目 | 確認内容 |
|---------|---------|
| リポジトリの公開元 | 誰が作ったものか。信頼できるソースか |
| 中身の確認 | 何をしているか全て読んで理解できるか |
| 依存関係 | 外部パッケージへの依存はないか。ある場合は安全か |
| 実行内容 | 外部通信、ファイル操作、環境変数参照など、意図しない動作がないか |
| `npx` 等の利用 | 任意コードの実行リスクがないか |
| サンドボックス検証 | 本番環境に入れる前に検証環境でテスト済みか |
| PM承認 | PMが内容を確認し、採用を承認したか |

**採用済みSkillsは「承認済みSkills一覧」で管理する。** 採用理由・審査日・PM承認日を記録する。
外部Skillsの自動探索・自動導入は初期段階では行わない。

#### 32. skills：コードレビュー（PMレビュー補助基盤）【新設】

**役割：** レビューの抜け漏れを減らし、若手案件でもレビュー観点を一定化する。人間レビューを代替するのではなく、**PMレビューを補助する基盤**。

**対象観点：**
- 仕様一致：仕様書通りに動作するか
- 可読性：コードが読みやすいか
- 命名：変数名・関数名が分かりやすいか
- 例外処理：異常系で適切にエラー処理されるか
- セキュリティ：基本的な脆弱性対策があるか
- テスト不足：テストされていない分岐やパスがないか
- 保守性：将来の改修時に理解・変更しやすいか
- パフォーマンス：初歩的な問題（N+1クエリ、不要なループ等）がないか

**使い方：**
- 若手がPMにレビュー依頼を出す前に、このSkillsでセルフチェックする
- PMがレビュー時に、観点の抜け漏れ確認として使う
- AIツールにレビュー補助として読み込ませ、指摘候補を出させることも可

#### 品質共通基盤の運用ルール

- これらの共通基盤は全案件で使用する（省略不可）
- PMが定期的にアップデートし、案件で得た知見を反映して育てていく
- 新しい案件で発覚した問題点は、共通基盤に還元してナレッジを蓄積する
- 若手が増えても、この基盤があれば同じ品質水準で開発できる体制を目指す
- **外部Skillsの導入は必ず審査ルールに従う（セキュリティチェック参照）**

**これがこのサービスの品質の根幹。**
個人のスキルではなく、仕組みで品質を担保する。
**最終的な品質責任はPMが負う。** 共通基盤はPMの判断を支える仕組みであり、PMのレビュー・判断を代替するものではない。

#### 将来の拡張候補

以下は現段階では導入しないが、運用が安定した後の拡張候補として記録しておく。

| 候補 | 内容 | 導入条件 |
|------|------|---------|
| **Hooks** | コミット時・PR作成時に自動チェックを実行するライフサイクル自動化 | CLAUDE.md / rules / skills の運用が安定し、自動化で効率化できる箇所が明確になった後 |
| **Orchestration** | 複数のAIエージェントを並列で統制し、開発・テスト・レビューを同時進行させる | 案件数が増え、手動のレビューフローがボトルネックになった後 |

まずは CLAUDE.md / .claude/rules/ / skills の運用を固めることを優先する。

### 契約形態の考え方

工程によって契約の性質が異なるため、以下のように使い分ける。

| 工程 | 契約性質 | 理由 |
|------|---------|------|
| 要件整理パック（Phase 1〜3の一部） | **準委任** | 成果物の完成を約束するのではなく、整理・助言を行う業務 |

**要件整理パックの詳細定義：**

| 項目 | 内容 |
|------|------|
| 成果物 | ヒアリング議事録、要件整理書（機能一覧・画面一覧・データ構造の概要）、概算見積もり |
| レビュー回数 | お客様との確認ミーティング最大2回まで含む。3回目以降は別途費用 |
| 完了の定義 | 成果物を納品し、確認ミーティング（1回）を実施した時点で完了とする。納品後5営業日以内に修正依頼がない場合も完了とみなす |
| 開発への移行 | 開発に進む場合は別途開発契約（請負）を締結する。要件整理パックの費用は開発費用に充当可 |
| 開発に進まない場合 | 成果物はお客様に帰属する。他社への引き継ぎに使用して構わない |
| 開発・テスト・納品（Phase 3後半〜6） | **請負** | 仕様書に基づく成果物の完成責任を負う |
| 保守運用（Phase 7） | **準委任** | 時間ベースの継続的な対応業務 |

**運用方法：**
- 基本契約書の中に請負・準委任の両方の条項を用意しておく
- 個別契約書（発注書）で案件ごとにどちらを適用するか明記する
- スモール案件で要件整理〜開発を一括受注する場合は請負に寄せてもよいが、その場合は仕様凍結後に開発着手という流れを厳守する

## 業務フロー全体像

```
Phase 0：問い合わせ対応
  ↓
Phase 1：ヒアリング・スコープ確認
  ↓
Phase 2：見積もり・契約
  ↓
Phase 3：要件定義・仕様策定
  ↓
Phase 4：開発
  ↓
Phase 5：テスト
  ↓
Phase 6：納品・検収
  ↓
Phase 7：保守運用（オプション）
```

## Phase 0：問い合わせ対応

### 目的
問い合わせの内容を確認し、対応可否を判断する。

### 担当
PM

### 作業内容

1. 問い合わせ内容を確認する
2. 以下の対応可否を判断する
   - 得意領域（Excel置換、管理画面、SaaS連携ツール）に該当するか
   - 対応外の案件（基幹系、ネイティブアプリ、法規制対応等）ではないか
   - 明らかに予算が合わない案件ではないか
3. 対応可能であればヒアリング日程を調整する
4. 対応外であればその旨を丁寧に伝え、断る

### 判断基準：この段階で断る案件

**技術・領域として対応外：**
- 会計・給与・税務計算等の基幹系
- 医療情報・金融取引等の法規制対応が必要な領域
- 24時間365日の高可用性が前提の大規模サービス
- iOS/Androidネイティブアプリ開発
- 機械学習モデルの独自開発

**進行不能・利益が出ない案件：**
- 「とにかく急ぎで、仕様は後から」という案件
- 業務フロー整理に協力いただけない案件
- 意思決定者が不在、または担当者に決定権がない
- 現行業務が未整理で、ヒアリングに必要な情報を出せない
- 予算・納期・要求のバランスが明らかに崩れている
- 無償提案（コンペ等）を過度に求める案件
- 必要なコミュニケーション頻度に応じていただけない

### 所要時間目安
問い合わせ確認〜返信：当日〜翌営業日

### 使用ドキュメント
なし（メール/チャットでのやりとり）

## Phase 1：ヒアリング・スコープ確認

### 目的
お客様の業務課題と「作りたいもの」を把握し、対応範囲（スコープ）の概要を掴む。

### 担当
PM主導（若手は同席し、議事録作成・資料整理・論点整理を主担当で行う）

### NDA締結のタイミング

初回ヒアリングの段階では機密情報を出さない前提で進める。
NDAの締結は以下のタイミングで行う。

- **初回ヒアリング**：機密情報なしで業務課題・作りたいものの概要を確認する。NDAは不要。
- **詳細資料共有・実データ提示が必要になった段階**：NDAを締結してから進める。
- **遅くともPhase 2（契約締結）までには必ず締結する。**
- お客様指定のNDAが来る場合は、内容を確認のうえ対応する。

### 作業内容

1. **ヒアリング実施（30分〜1時間、オンライン）**
   - ヒアリングシートに沿って聞く
   - 現在の業務フロー（As-Is）を確認する
   - 困っていること・課題を深掘りする
   - 作りたいもののイメージを確認する
   - 利用者・利用頻度・扱うデータの種類を確認する
   - 既存ツール（Excel、SaaS等）の有無を確認する
   - 予算感・希望納期を確認する
   - 機密データの有無を確認する

2. **ヒアリング後の社内整理（PM + 若手）**
   - ヒアリング議事録を整理する
   - 必要な画面・機能の概要を洗い出す
   - 対応範囲（スコープ）と対応外を整理する
   - **見積ルール表に基づいて**概算工数を見積もる
   - プランの判定を行う（スモール / スタンダード / カスタム）

3. **スコープ確認の共有（PM → お客様）**
   - ヒアリング内容のまとめをお客様に共有し、認識に齟齬がないか確認する

### ゲート判定（Phase 1 → Phase 2に進むか）

以下が全て確認できたら次のPhaseに進む。

- [ ] お客様の課題が明確になっている
- [ ] 作るものの概要が掴めている
- [ ] 対応範囲と対応外が整理できている
- [ ] 見積ルール表に基づいて概算工数が見積もれている
- [ ] お客様の予算感と大きく乖離していない

ヒアリング：30分〜1時間
社内整理：2〜4時間

## 見積ルール

概算見積もりの精度を安定させるため、以下の基準で工数を算出する。
案件を重ねるたびに実績値で補正していく。

### 画面単位の基準工数（若手1名あたり）

| 画面の種類 | 基準工数 | 備考 |
|-----------|---------|------|
| 単純な入力フォーム（CRUD） | 0.5〜1日 | 登録・編集・削除のみ |
| 一覧画面（検索・絞り込み付き） | 1〜2日 | ソート・ページネーション含む |
| ダッシュボード・集計画面 | 2〜3日 | グラフ・集計ロジック含む |
| CSV入出力付き画面 | +0.5〜1日 | 上記に加算 |

### 加算要素

| 要素 | 加算工数 |
|------|---------|
| ユーザー権限管理（ロール分離） | +1〜3日 |
| 外部サービス連携（API 1つ） | +1〜3日 |
| メール・チャット通知 | +0.5〜1日 |
| 承認フロー | +1〜2日 |
| データ移行（既存Excel等からの取り込み） | +1〜3日 |

### 共通係数

| 工程 | 開発工数に対する比率 |
|------|-------------------|
| 要件定義・仕様書作成（PM） | 開発工数の30〜50% |
| テスト・セキュリティチェック | 開発工数の20〜30% |
| ドキュメント作成（マニュアル等） | 開発工数の10〜15% |
| PMレビュー・管理 | 開発工数の15〜20% |
| 予備バッファ | 全体工数の15〜20% |

### 見積算出の手順

1. 画面一覧を作り、各画面の基準工数を積み上げる（若手工数）
2. 加算要素を足す
3. 合計に共通係数を掛けて全体工数を出す（PM工数は別枠で積む）
4. 予備バッファを加える
5. 若手工数 × 若手単価 + PM工数 × PM単価 で人件費を算出する
6. AIツール費・インセンティブ（売上の5%: 営業2% + PM3%）を加味して案件価格を決定する
7. 案件分類（テンプレ型 / 個別対応）を確認し、テンプレ型に該当しない案件は安易に値下げしない

### 内部単価表（社外非公開）

見積金額の算出と採算管理に使用する。対外的には公開しない。

**価格設計の原則：**
- 価格は「画面数」だけで決めない
- **若手開発工数・PM工数・顧客調整工数・テスト工数・納品工数**を分けて積み上げる
- 社内の基準単価と、対外的な案件価格は別物。案件価格は難易度・PM関与度・テンプレ適合度で決める
- テンプレ型案件と個別対応案件で価格思想を変える（「案件分類の考え方」を参照）

| 役割 | 社内基準単価 | 備考 |
|------|------------|------|
| 若手（開発・テスト・ドキュメント） | **40万円/人月**（約2万円/人日） | SES水準30〜35万に対して十分に良い水準。これを初期の採算ラインとする |
| PM（要件定義・レビュー・管理・顧客折衝） | ◯万円/人日 | 若手単価とは別枠で積む。PM工数込みで40万円と考えないこと |

| 項目 | 基準 |
|------|------|
| 最低受注金額 | ◯万円（これを下回る案件は受けない。PM工数だけで赤字になる案件を防ぐ） |
| 粗利率の下限 | ◯%（若手人件費・AIツール費・インセンティブを差し引いた後） |
| 保守超過時の追加単価 | ◯万円/時間 |

**この表は案件を重ねるたびに実績値で更新する。**
最初の3案件は実績データ収集期間と位置づけ、3案件終了後に単価と最低受注金額を見直す。

**注意：若手が月2〜3件回せる可能性はあるが、それはテンプレ型案件に限る。**
外部連携・複雑権限・曖昧要件・顧客確認遅延がある個別対応案件は高回転前提にしない。

## 支払条件ポリシー

### 基本方針

**「先にもらう」を基本設計とする。**

理由：
- ヒアリング・要件整理・仕様化にも工数がかかっており、これらを後払いだけにすると仕様だけ取られて流れるリスクがある
- 検収後100%払いは検収引き延ばしで資金繰りが悪化する
- 着手金なしで始めると「まだ払ってないけど始めて」が起きる

### 案件パターン別の支払条件

| パターン | 支払い方法 | 着手条件 |
|---------|-----------|---------|
| **要件整理パック**（5〜10万） | 契約時100%前払い | 入金確認後に着手 |
| **スモール開発**（15〜30万） | 契約時50% + 検収完了後50% | 着手金の入金確認後に着手 |
| **スタンダード開発**（30〜80万） | 契約時50% + 検収完了後50% | 着手金の入金確認後に着手 |
| **カスタム開発**（個別見積） | 要件定義を別契約で先行。開発契約時50% + 中間30% + 検収完了後20% | 各フェーズの入金確認後に次フェーズ着手 |
| **保守**（月額） | 月末締め翌月末払い | 保守契約締結後に開始 |

### 着手条件（これを満たすまで作業を開始しない）

以下の全てが揃った時点で着手する。

1. 契約書（基本契約 + 個別契約）が締結されている
2. 着手金の入金が確認できている
3. 必要資料をお客様から受領している

**「契約前に少しだけ見てほしい」「入金前だけど先に始めてほしい」は原則受けない。**

### 契約書・見積書・発注書に必ず明記する金額まわりの項目

| 項目 | 記載内容 |
|------|---------|
| 報酬額 | 総額、税別/税込の明記、追加費用が発生する条件 |
| 支払条件 | 前払い/分割/検収後払いの区分、各回の金額または比率 |
| 支払期日 | 請求書発行後◯営業日以内（例：30日以内）。関連法令を踏まえ、受領日から60日以内で適切に設定する ※具体的な法令適用は弁護士に確認 |
| 着手条件 | 着手金入金確認後、必要資料受領後、契約締結完了後 |
| 検収条件 | 検収期間、指摘方法、みなし検収の起算条件 |
| 追加作業の扱い | 仕様外は別見積、追加開発は変更合意書で合意後に着手 |
| 中途解約時の精算 | どの時点でいくら発生するか |

**見積書と契約書で支払条件がズレないように必ず一致させる。**

### 中途解約時の精算ルール

お客様都合で案件が中途解約となった場合の精算基準。
**実施済み業務相当分は精算対象とし、未実施分は契約条件に従い返金有無を判断する。**

| 解約のタイミング | 精算ルール |
|----------------|-----------|
| 契約締結後〜要件定義着手前 | 着手金は原則返金しない。ただし実質未着手の場合は協議のうえ対応する |
| 要件定義中 | 実施済みの業務相当分を精算する。未実施分があれば協議のうえ返金を検討する |
| 仕様凍結後〜開発中 | 実施済みの業務相当分（要件定義＋開発進捗分）を精算する。仕様書は成果物としてお客様に引き渡す |
| テスト中〜納品前 | 全額を精算対象とする。成果物（コード・仕様書・テスト結果）は引き渡す |
| 検収期間中 | 納品済みのため、残金の支払い義務が発生する |

**精算の具体的な条件は契約書に明記する。弁護士レビューを推奨。**

### 「軽微な修正」と「追加開発」の境界

お金が崩れる最大の原因はここ。明確に分ける。

| 区分 | 定義 | 費用 |
|------|------|------|
| 不具合修正 | 仕様書に記載された内容が正しく動かない | 無償（契約不適合責任の範囲） |
| 軽微な修正 | 仕様書の変更を伴わない微調整（文言修正、色の変更等） | 納品後1週間以内は無償。以降は保守契約の範囲 |
| 追加開発 | 仕様書の変更を伴う機能追加・仕様変更 | 別途見積もり。変更合意書で合意後に着手 |

## Phase 4：開発

仕様書に基づいてシステムを構築する。

### 担当
若手が実務オーナー、PMはレビュー依頼に対してレビューする

### 作業内容

1. **開発環境のセットアップ（若手）**
   - **【必須確認】リポジトリの場所・作成の確認**：開発着手前に、PM（または作業指示者）に以下を必ず確認する。確認が取れるまで開発に着手しない
     - Gitリポジトリをどのディレクトリに作成するか
     - GitHubリポジトリを作成済みか。未作成の場合、作成してよいか
     - GitHubリポジトリ名の確認
   - リポジトリの作成（案件ごとに1リポジトリ、private設定、命名規則に従う。Gitリポジトリ管理ルール参照）
   - 開発環境の構築
   - AIツール（Claude Code / Cursor等）の準備
   - **CLAUDE.md + .claude/rules/ の配置**：プロジェクトルートに配置し、AIへの指示ルール・コード規約を適用する
   - **skills（デザイン）の適用**：UI/UXの標準ルールをプロジェクトに組み込む

2. **開発の実施（若手が主体的に進める）**
   - 仕様書の機能単位で開発を進める
   - 1機能ごとにAIに指示 → コード生成 → 動作確認のサイクルを回す
   - **CLAUDE.md + rules を通してAIに指示する**（直接の自由な指示ではなく、共通基盤のルールに沿った開発を行う）
   - 仕様書に記載のない機能は作らない（勝手に判断しない）
   - 仕様の解釈に迷ったらPMに相談する（ただし、まず自分で仕様書を読み返すこと）

3. **PMへのレビュー依頼（若手 → PM）**
   - 若手が開発の区切りでPMにレビューを依頼する
   - **レビュー依頼前に、skills（コードレビュー）でセルフチェックを行う**
   - **レビュー単位は案件規模に応じて柔軟に設定する：**
     - スモール案件：画面単位または全体完成時にまとめてレビュー
     - スタンダード案件：重要機能単位または開発区切り単位でレビュー
     - カスタム案件：PM判断で適切な粒度を設定する
   - PMは随時伴走するのではなく、**レビュー依頼が上がったタイミングでレビューする**

4. **PMによるレビュー（レビュー観点一覧に基づく）**
   - **以下の標準レビュー観点を必ず確認する：**
     - 仕様一致：仕様書通りに動作するか
     - 入力バリデーション：不正入力を弾けるか
     - 権限制御：権限外の操作ができないか（権限管理がある場合）
     - エラーハンドリング：異常系で適切にエラー処理されるか
     - セキュリティ：SQLインジェクション、XSS、CSRF等の基本対策があるか
     - 命名・可読性：変数名・関数名が分かりやすいか、コードが読みやすいか
     - ログ出力：必要な箇所にログが出力されているか
   - 問題があれば修正を指示する

4. **中間確認（以下の条件に該当する場合は必須）**
   - **必須条件：** 画面数5以上 / 外部サービス連携あり / 権限管理あり / 仕様変更リスクが高いUI中心の案件
   - 上記に該当しないスモール案件では任意（PM判断）
   - **中間確認は、仕様凍結済み内容との整合確認を目的とする**
   - 中間確認で出た新規要望は変更管理に回す（その場で受けない）
   - 仕様確認なのか、UI調整なのか、追加要望なのかをPMがその場で切り分ける

### 若手向けルール（自分の案件として進める）

- 自分の担当案件は自分が進める。PMに指示を待つのではなく、自分で考えて動く
- 仕様書に書いてないことを「良かれと思って」追加しない
- AIが生成したコードは必ず動作確認してからコミットする
- 仕様の解釈に迷ったら、まず仕様書を読み返す。それでも分からなければPMに相談する
- エラーが出たら自分で30分調べて解決しない場合はPMに相談する
- 進捗・課題・不明点を自分から整理してPMに上げる（1日の終わりにチャット or 日報）
- お客様のデータをAIに直接入力しない（情報取扱ルールを遵守）
- 顧客への通常連絡（日程調整、確認依頼等）は自分で対応する。判断が必要な内容はPMに相談してから送る

### ゲート判定（Phase 4 → Phase 5に進むか）

- [ ] 仕様書の全機能が実装されている
- [ ] PMのコードレビュー（レビュー観点一覧に基づく）を通過している
- [ ] 明らかなバグが残っていない
- [ ] 開発環境で一通り動作することが確認できている

### 所要時間目安
スモール：3〜7日
スタンダード：7〜20日
カスタム：案件による

### 使用ドキュメント
- 機能仕様書（Phase 3で作成済み）
- **レビュー観点一覧**（★作成必要）
- **社内開発ガイドライン**（★作成必要）

## Phase 5：テスト

### 目的
仕様書通りに動くことを確認する。品質の証拠を残す。

### 担当
**若手が主体的に進める。** テスト仕様書の作成・テスト実施・証跡整理は若手が主担当。PMはテスト仕様書のレビュー・抜け漏れ確認・品質の最終判断を行う。

※立ち上げ初期はPMがテスト観点の洗い出しを主導するが、若手が自力で回せる状態を目指す。

### 作業内容

1. **テスト観点の洗い出し（初期はPM主導 → 段階的に若手に移行）**
   - 機能仕様書の各項目からテスト観点を洗い出す
   - **skills（テスト）のモジュールを適用**：`test-spec-creator` で仕様書からテスト観点を起こし、`boundary-and-validation-test` で境界値・バリデーション観点を補強する
   - 以下の観点を必ず含める：
     - 正常系（基本操作）
     - 異常系（不正入力、未入力、型不一致）
     - 境界値（最大値、最小値、上限）
     - 権限違い（権限がない操作を試みた場合）
     - 重複（同一データの二重登録）
     - 画面遷移（想定外の遷移、ブラウザバック）

2. **テスト仕様書の作成（若手、PMレビュー）**
   - PMが洗い出したテスト観点をもとにテスト仕様書を作成する
   - PMが項目レベルでレビューし、漏れがないか確認する

3. **テストの実施（若手）**
   - テスト仕様書に沿って1項目ずつ実施する
   - 結果をOK / NGで記録する
   - NGの場合は不具合内容を記録し、修正 → 再テストする

4. **セキュリティチェックの実施（若手、PMレビュー）**
   - **skills（セキュリティチェック）を適用**：標準チェックリストに沿って確認する
   - 結果をOK / NGで記録する

5. **テスト結果のレビュー（PM）**
   - 全テスト項目がOKであることを確認する
   - テスト漏れがないか仕様書と突合する
   - 重大な不具合が残っていないことを確認する

6. **リリース判定（PM）**
   - **リリース判定チェックリスト**に沿って最終確認する
   - 受入テスト環境（ステージング環境）を用意する

### ゲート判定（Phase 5 → Phase 6に進むか）

- [ ] テスト仕様書の全項目がOKである
- [ ] セキュリティチェックの全項目がOKである
- [ ] PMがテスト結果・セキュリティチェック結果をレビューし承認している
- [ ] 既知の不具合が全て対応済みである
- [ ] リリース判定チェックリストが全項目OKである
- [ ] 受入テスト環境が準備できている

### 所要時間目安
スモール：1〜3日
スタンダード：3〜5日
カスタム：案件による

### 使用ドキュメント
- **テスト仕様書**（★テンプレート作成必要）
- **リリース判定チェックリスト**（★作成必要）

## Phase 6：納品・検収

### 目的
成果物をお客様に引き渡し、合意のもと検収を完了する。

### 担当
若手が実務を主体的に進める。PMは仕様判断・追加要望判定・重大不具合判断・検収条件の最終判断を担う。

### 作業内容

1. **納品物の準備（若手が主担当、PMが最終確認）**
   - 要件定義書（版数・承認日入り）
   - 機能仕様書（版数・承認日入り）
   - ソースコード一式（Gitタグまたはコミットハッシュで版を特定）
   - テスト仕様書・テスト結果
   - セキュリティチェック結果
   - 簡易運用マニュアル
   - 環境情報（サーバー、DB、環境変数等）
   - **納品物一覧に版数・作成日を記載する**

**ソースコードの納品方法（標準）：**

| 方法 | 使用条件 |
|------|---------|
| **GitHubリポジトリの移管**（標準） | お客様がGitHub組織を持っている場合。リポジトリのownerをお客様に移管する |
| **privateリポジトリへの招待** | お客様がGitHubアカウントを持っているがリポジトリ移管が難しい場合 |
| **ZIPファイルで納品** | お客様がGitを使わない場合。クラウドストレージ経由で送付する |

原則はGitHubリポジトリの移管とする。方法はヒアリング時にお客様の環境を確認して決める。
案件ごとにリポジトリを切っているため、そのまま移管できる（Gitリポジトリ管理ルール参照）。
ドキュメント類（仕様書・マニュアル等）はコードとは別にPDFまたはMarkdownで納品する。

2. **納品・説明ミーティング（若手が主体、PMは必要に応じて同席）**
   - 若手が納品物一式を引き渡す
   - 若手がシステムの操作説明を行う（オンライン30分〜1時間）
   - 検収ルールの説明はPMが行う（または若手がPM確認済みの内容で説明する）

3. **お客様による受入テスト（お客様）**
   - お客様が実際に操作して確認する
   - 指摘事項は**受入テスト指摘票**で受け付ける

4. **指摘事項への対応（若手が取りまとめ・修正、PMは判断のみ）**
   - **若手が指摘を取りまとめ、以下に分類する：**
     - **P1（重大）**：主要機能が動かない、データが壊れる → 即時対応
     - **P2（中程度）**：一部機能の不具合、表示崩れ → 検収期間内に対応
     - **P3（軽微）**：文言修正、微細なUI調整 → 別紙管理のうえ検収完了可
   - 仕様範囲内の不具合（P1/P2）：無償で修正する
   - 仕様範囲外の追加要望：仕様変更として別途見積もりを提示する
   - **仕様書が判断の根拠。曖昧な場合はPMが判断し記録を残す。**

5. **検収完了**
   - P1/P2の不具合が全て解消されていることを確認する
   - P3の軽微不具合は別紙管理リストに記載のうえ、検収完了とすることがある（お客様合意のうえ）
   - お客様から検収完了の連絡を受ける（検収書に署名・捺印）
   - 残金の請求書を発行する

6. **社内クロージング**
   - **顧客データ削除チェックリスト**に沿って、開発環境からお客様データを削除する
   - プロジェクト完了報告を社内で行う（工数実績の記録）
   - 事例化の許可をお客様に確認する
   - **案件クロージングチェックリスト**で漏れがないか最終確認する

### 検収ルール（契約書・個別契約書にも明記する）

| 項目 | ルール |
|------|-------|
| 検収期間 | 納品後7営業日以内 |
| 指摘方法 | 受入テスト指摘票（メール添付 or 共有シート）で受け付ける |
| 指摘の分類 | P1（重大）/ P2（中程度）/ P3（軽微）で重大度を付ける |
| 検収合格条件 | P1/P2が全て解消されていること |
| P3の扱い | 別紙管理リストに記載のうえ、検収完了可（双方合意） |
| みなし検収 | 検収期間内にお客様から指摘がない場合は検収完了とみなす |
| 検収不合格時 | 不具合修正後に再納品し、**修正対象範囲に限定して**3営業日の再検収期間を設ける |
| 判断基準 | 仕様書に記載された内容が動くこと |

### ゲート判定（Phase 6完了）

- [ ] 納品物が全て揃っている（版数・作成日記録済み）
- [ ] お客様に納品物を引き渡し済みである
- [ ] P1/P2不具合が全て解消されている
- [ ] 検収書が受領できている
- [ ] 残金の請求書を発行済みである
- [ ] 顧客データ削除チェックリストが完了している
- [ ] 案件クロージングチェックリストが完了している

納品準備：1〜2日
検収期間：7営業日
指摘対応：0〜5日

## Phase 7：保守運用（オプション）

### 目的
納品後のシステムを安定運用し、継続的に改善する。

### 担当
若手が一次対応の実務オーナー。PMはエスカレーション判断・追加開発判断・契約範囲判断に寄せる。

### 保守プラン定義

| | ライト | スタンダード | プレミアム |
|---|--------|------------|----------|
| **月額（税別）** | 2万円 | 5万円 | 10万円〜 |
| **月間対応時間上限** | 30分 | 2時間 | 5時間 |
| **対応対象** | 問い合わせ対応・障害一次切り分け | 左記＋軽微な改修・バグ修正 | 左記＋機能追加・改善対応 |
| **受付時間** | 営業日 10:00〜18:00 | 営業日 10:00〜18:00 | 土日含む 10:00〜18:00 |
| **初回応答目安** | 翌営業日 | 当日中 | 2時間以内 |
| **月次レポート** | 簡易稼働レポート | 改善提案付きレポート | 改善提案付きレポート＋月1回ミーティング |
| **超過時の追加単価** | ◯万円/時間 | ◯万円/時間 | ◯万円/時間 |

※超過が見込まれる場合はPMが事前にお客様に通知し、合意のうえで対応する。
※仕様書の変更を伴う機能追加は、保守ではなく追加開発として別途見積もる。

### 保守対応ルール

| 項目 | ルール |
|------|-------|
| 問い合わせ手段 | メール or チャット（電話は原則不可、記録が残らないため） |
| 受付時間 | 営業日 10:00〜18:00（プレミアムは土日含む） |
| 初回応答目安 | ライト：翌営業日 / スタンダード：当日中 / プレミアム：2時間以内 |
| 問い合わせの分類 | 障害（システムが動かない）/ 質問（使い方）/ 要望（機能追加・改善） |
| 緊急度 | 高（業務停止）/ 中（一部機能に影響）/ 低（業務への影響なし） |
| 月次レポート提出日 | 翌月5営業日以内 |
| 契約時間超過時 | PMがお客様に事前通知し、追加費用を合意してから対応 |

### 作業内容

1. **保守契約の締結（PM）**
   - 検収完了時に保守契約の要否を確認する
   - 保守プラン（ライト / スタンダード / プレミアム）を提案する
   - 保守契約書を締結する（準委任型）

2. **月次の保守対応（若手 + PM）**
   - お客様からの問い合わせに対応する
   - 障害 / 質問 / 要望を切り分けて対応する
   - バグ修正・軽微な改修を実施する
   - 対応内容と対応時間を保守対応ログに記録する
   - 契約時間の上限を超えそうな場合はPMがお客様に事前連絡する

3. **月次レポートの作成・送付（若手が作成、PMがレビュー）**
   - 月間の対応内容と対応時間をまとめる
   - 残り対応可能時間を記載する
   - 改善提案があれば記載する

4. **追加開発の受注（PM）**
   - 保守対応の中で追加開発のニーズが出た場合は、別案件として見積もる
   - 保守の時間枠内で対応できない規模のものは、新規案件として扱う

### 保守で揉めないためのルール

- 対応範囲は「時間」で管理する（回数ではない）
- 「軽微な改修」の定義：仕様書の変更を伴わない修正
- 仕様書の変更を伴う追加機能は保守ではなく追加開発
- 毎月の対応時間を記録し、お客様にも月次レポートで共有する

### 使用ドキュメント
- **保守契約書**（★テンプレート作成必要 → 弁護士レビュー推奨）
- **月次保守レポート**（★テンプレート作成必要）
- **保守対応ログ**（対応日時・内容・対応時間を記録するシート）

## 仕様変更が発生した場合のフロー

開発途中または検収中に、当初の仕様にない要望が出た場合のルール。

```
お客様から変更・追加の要望
  ↓
PMが受け取る
  ↓
仕様書と照合し、仕様範囲内か範囲外か判断
  ↓
【仕様範囲内】→ 不具合として無償対応
【仕様範囲外】→ 変更管理票を起票し、以下を実施
  ↓
影響範囲と追加工数を見積もる
  ↓
追加費用・納期変更をお客様に提示
  ↓
合意 → 変更合意書を取り交わし、仕様書を更新して開発（版数を上げる）
不合意 → 現行仕様のまま進行
```

### 変更管理票の項目

| 項目 | 内容 |
|------|------|
| 変更要求ID | 連番で管理（例：CHG-001） |
| 起票日 | |
| 起票者 | お客様の誰が要望したか |
| 変更内容 | 何をどう変えたいか |
| 仕様範囲内/外の判定 | PMが仕様書と照合して判定 |
| 判定根拠 | 仕様書のどこに基づくか |
| 影響範囲 | どの画面・機能に影響するか |
| 追加工数 | 見積ルール表に基づく |
| 追加費用 | |
| 納期影響 | |
| お客様承認日 | |
| 変更合意書締結日 | 追加費用が発生する場合は変更合意書を取り交わす |
| 対応完了日 | |
| ステータス | 起票 / 見積提示中 / 承認済 / 対応中 / 完了 / 却下 |

### 仕様変更の判断基準

- 仕様書に明記されている → 仕様範囲内（不具合対応）
- 仕様書に明記されていない → 仕様範囲外（変更管理票を起票）
- 「当然含まれると思っていた」は仕様書が判断基準

**これが仕様書を先に作る最大の理由。**

## 情報セキュリティ・データ取り扱いルール

### データの取り扱い

| 項目 | ルール |
|------|-------|
| 実データの利用 | 原則使用しない。ダミーデータまたはマスキングデータを使用する |
| やむを得ず実データを使う場合 | NDA締結済みであること。PM承認のうえ、利用後は速やかに削除する |
| 顧客ごとのデータ保存場所 | 案件ごとに分離したフォルダ/リポジトリで管理する。他案件と混在させない |
| ローカル保存 | 開発に必要な範囲で可。ただし案件完了後に削除する |
| 私物端末 | 会社支給端末を基本とする。支給が間に合わない場合は、PM承認のうえ私物端末を暫定利用可とする。その場合はディスク暗号化・パスワード設定・OSの自動アップデートを必須とする。支給端末が用意でき次第移行する |

### AIツール利用ルール

| 項目 | ルール |
|------|-------|
| お客様の機密データ | AIに直接入力しない。これが最重要ルール。「入れてから消す」ではなく「そもそも入れない」を徹底する |
| AIへの指示に含めてよい情報 | データ構造、ロジック、コード規約。固有名詞・個人情報は含めない |
| 利用するAIサービス | API経由を優先する（学習に使用されないプランを選択する）。利用サービスをリスト化し、PMが承認したもののみ使用可 |
| やむを得ず機密性のある情報を扱う場合 | PM承認のうえ、API経由（学習オフ）のサービスのみ使用する。利用後は利用範囲を記録する |

### アクセス権限管理

| 項目 | ルール |
|------|-------|
| Gitリポジトリ | private設定。案件ごとに作成し、関係者のみアクセス可 |
| 開発環境・ステージング環境 | 案件メンバーのみアクセス可 |
| 退職・離任時 | 即日で全アクセス権限を剥奪する。PMが確認する |
| お客様環境へのアクセス | 案件完了後に返却。アクセスキー等は削除する |

### Gitリポジトリ管理ルール

**原則：案件ごとにリポジトリを切る。顧客ごとにまとめない。**

| 項目 | ルール |
|------|-------|
| 作成単位 | **案件ごとに1リポジトリ**。同じ顧客の別案件でもリポジトリは分ける |
| 公開設定 | 必ずprivate |
| 命名規則 | `{顧客名}-{案件概要}-{年月}`（例：`tanaka-shoji-inventory-202604`） |
| アクセス権 | 案件担当メンバーのみ招待する。他案件の若手には付与しない |
| 納品時 | リポジトリのownerをお客様に移管する（標準の納品方法） |
| 納品後 | 移管済みリポジトリから当社メンバーのアクセスを外す。保守契約がある場合のみアクセスを維持する |
| アーカイブ | 納品・検収完了後、当社側の控えが必要な場合はアーカイブ設定にする |

**案件ごとに切る理由：**
- 納品時にリポジトリをそのままお客様に移管できる（切り出し作業が不要）
- 案件完了後のデータ削除チェックで「この案件のデータは全部消した」と言い切れる
- アクセス権限を案件単位で管理でき、担当外の若手に別案件のコードが見えない
- 同じ顧客の1案件目が納品済みでアクセスを閉じるべきとき、2案件目と分離できる

### 納品後のデータ処理

案件完了後、**顧客データ削除チェックリスト**に沿って以下を実施する。

- [ ] ローカル環境のお客様データを削除した
- [ ] 開発環境・ステージング環境のお客様データを削除した
- [ ] AIツールに機密データを入力していないことをメンバーに確認した（AI利用ルール遵守の確認）
- [ ] お客様から預かった資料・データを返却または削除した
- [ ] Gitリポジトリのアクセス権限を整理した（移管済みの場合は当社メンバーのアクセスを外す。保守契約がある場合のみ維持。控えが必要な場合はアーカイブ設定）
- [ ] PMが削除完了を確認した

### 監査証跡（記録として残すもの）

ルールを守っている証拠を記録として残す。万が一のトラブル時にも説明責任を果たせるようにする。

| 記録対象 | 記録方法 | 保管場所 |
|---------|---------|---------|
| PM承認済みAIサービス一覧 | 利用開始時にPMがリスト化し、更新のたびに版数を上げる | 社内共有フォルダ |
| 顧客データ削除チェック結果 | 上記チェックリストの実施日・実施者・PM確認日を記録する | 案件フォルダ |
| アクセス権の付与・剥奪履歴 | 誰に・いつ・どのリポジトリ/環境のアクセス権を付与/剥奪したかを記録する | 案件フォルダ |
| 私物端末暫定利用の承認記録 | 利用者・端末情報・PM承認日・セキュリティ設定確認日を記録する | 社内共有フォルダ |
| NDA/契約書の締結記録 | 締結日・対象者・契約書のPDFを保管する | 案件フォルダ |

**最初から完璧にやる必要はないが、「記録を残す習慣」を初案件から始める。**
