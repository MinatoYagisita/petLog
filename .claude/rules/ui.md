# UI/UX ルール

> 前提: React + Tailwind CSS / Next.js

## 基本姿勢

- 業務ツールの UI は「派手さ」より「使いやすさ」を重視する
- 毎日何十回も操作する画面であることを前提に設計する
- 迷ったら「業務担当者が迷わず操作できるか」で判断する

## Tailwind CSS の利用方針

- ユーティリティクラスを直接使うのが基本。カスタム CSS は最小限にする
- 余白・サイズは Tailwind のスケール（`p-4`, `gap-6`, `text-sm` 等）に揃える
- レスポンシブは `sm:` / `md:` / `lg:` プレフィックスで対応する
- 色はプロジェクト内で定義したカラーパレット（`tailwind.config.ts`）を使う
- 長いクラス列は改行して読みやすくする。必要に応じて `cn()` ユーティリティで結合する
- `@apply` の多用は避ける（Tailwind の利点が失われる）

```tsx
// 崩れにくい書き方
<div className="flex flex-col gap-4 p-6 max-w-2xl mx-auto">
  <h1 className="text-xl font-bold text-gray-900">ユーザー一覧</h1>
  {/* ... */}
</div>
```

## 一覧画面

- データ件数が多い場合はページネーションを実装する
- ソート・フィルタ機能を提供する（最低限、主要カラムでのソート）
- 一覧の各行で「何のデータか」が一目で分かるようにする
- 0件の場合は空白ではなく「該当データがありません」等のメッセージを表示する
- ローディング中は Skeleton（`animate-pulse`）で状態を示す
- 横スクロールが必要なテーブルでは ID 列等の固定を検討する（`sticky left-0`）
- 主要列が折り返しや省略（`truncate`）で読めなくなっていないか確認する
- 1画面内の情報量が多すぎないか確認する
- 行クリックとボタン操作の役割を明確にする（曖昧にしない）

## 検索 UI

- 検索条件が多い場合は、よく使う条件を上に配置する
- 検索ボタンとクリアボタンを用意する
- 検索結果が0件の場合は、条件の見直しを促すメッセージを表示する
- 検索条件は画面遷移後も保持する（URL パラメータ or state で管理）
- 検索条件が多い場合はグルーピングや折りたたみを検討する
- 初期表示時の検索条件が妥当か確認する
- 絞り込み結果の件数を表示する

## 入力フォーム

- 必須項目には `*` や「必須」ラベルを付ける
- 入力例（`placeholder`）を適切に設定する
- バリデーションエラーは該当項目の近くに表示する（`text-red-600 text-sm mt-1`）
- 入力途中でページを離れる場合は確認ダイアログを出す（`beforeunload`）
- 保存成功時はフィードバックを表示する（トーストなど）
- ラベル位置（上 / 左）を画面間で統一する
- 必須と任意の区別が一目で分かるようにする
- フォームが長い場合はセクション分けする（`<fieldset>` や見出しで区切る）
- 入力順が業務フローに沿っているか確認する
- 入力ミスしやすい項目には補助説明を付ける

## ボタン配置

- 主要アクション（保存、登録）は右下または右上に配置する
- 破壊的アクション（削除）は主要アクションと離して配置する
- 削除操作には確認ダイアログを必ず設ける
- ボタンのラベルは具体的にする（「OK」ではなく「保存する」「削除する」）
- 処理中はボタンを `disabled` にし、スピナーを表示して二重送信を防ぐ
- Primary / Secondary / Danger の役割が見た目（色・スタイル）で区別できるようにする
- 戻る・キャンセル・保存の配置ルールを画面間で統一する

## エラー表示

- エラーメッセージはユーザーが「次に何をすればよいか」分かる内容にする
- 技術的なエラーコードだけを表示しない
- 致命的エラーと入力エラーを視覚的に区別する（色、アイコン、配置）
- エラー発生時に入力済みデータが消えないようにする
- 複数項目エラー時に、どの項目を直せばいいか分かるようにする

```tsx
// 悪い例
<p>Error: 422 Unprocessable Entity</p>

// 良い例
<p className="text-red-600 text-sm mt-1">
  メールアドレスの形式が正しくありません。例: user@example.com
</p>
```

## 余白・レイアウト

- 余白は Tailwind のスペーシングスケールに揃える（`p-4`, `gap-6`, `mt-8` 等）
- 関連する要素はグルーピングし、グループ間に十分な余白を取る
- 画面端ぎりぎりまで要素を置かない（`px-4` や `max-w-7xl mx-auto`）
- 横幅が広すぎる一行テキストは `max-w-prose` を設定する
- 見出し・本文・補足の階層が視覚的に分かれているか確認する
- テーブルやフォーム内で文字が詰まりすぎていないか確認する

## タイポグラフィ

- Tailwind の `text-sm`（14px）/ `text-base`（16px）を本文に使う
- 見出し・本文・補足の3段階程度でサイズを使い分ける
- 行間は `leading-relaxed`（1.625）を目安にする
- 色だけで情報を伝えない（色覚多様性への配慮）

## 一貫性

- 同じ意味のボタンは同じラベル・同じ位置にする
- 日付フォーマットを画面ごとに変えない（`YYYY/MM/DD` など統一）
- ステータス表示のラベルや色を画面ごとに変えない
- 用語を統一する（「ユーザー」と「利用者」を混在させない）
- 同じ操作に同じアイコン・文言・色を使う
- モーダル・トースト・ダイアログの扱いを画面間で統一する

## アクセシビリティ

- 画像には `alt` テキストを設定する
- フォーム要素には `<label>` を紐づける（`htmlFor` を使う）
- キーボード操作でフォーカスが見えるようにする（`focus:ring-2 focus:ring-blue-500`）
- フォーカス順が自然か確認する（DOM 順と視覚的な順序を一致させる）
- モーダル表示時はフォーカストラップを実装する
- コントラスト比は WCAG AA 基準（4.5:1 以上）を目指す
- エラーを色だけで示さない（アイコンやテキストも併用する）
- エラーがスクリーンリーダーでも把握できる形（`aria-live="polite"` 等）にする
- プレースホルダーだけで意味を伝えない（入力後に消えて分からなくならないように）
