# ペットお薬手帳・健康管理アプリ 要件定義

---

## 1. プロダクト概要

### 1.1 背景

- ペットの投薬は時間・回数・種類が複雑でミスが起きやすい
- 健康記録、通院履歴、処方情報が分散しやすい
- 継続して使えるシンプルな管理ツールが必要

### 1.2 目的

- 投薬ミスを防ぐ（最優先）
- 健康状態を簡単に記録・把握できるようにする
- 通院・処方・記録を一元管理する

### 1.3 対象ユーザー

- 犬・猫などの飼い主
- 投薬や通院管理に不安がある人

### 1.4 提供価値

- ワンタップ投薬管理
- 入力負荷の低い健康記録
- 医療情報の一元管理

### 1.5 KPI

- 投薬チェック率：90%以上
- 継続率（7日）：60%以上
- 健康記録入力：週3回以上

---

## 2. 設計思想

- ユーザーは正確に使わない前提
- 不完全データでも価値を出す
- 継続性を最優先
- 後編集を前提とする

---

### 3.1 含む

- 認証（メール＋パスワード）
- ペット管理（複数対応）

- 投薬管理（残量管理・通知）
- 健康記録

### 3.2 含まない（将来拡張）

## 5. 機能要件

---

### 5.1 認証・会員登録

#### 会員登録

- メールアドレス＋パスワードで新規登録
- 登録後はペット登録画面へ自動遷移

#### ログイン

- メールアドレス＋パスワードでログイン
- ログイン状態を維持（セッション保持）
- 将来対応：Googleアカウントでのログイン

#### パスワードリセット

- メールアドレスにリセットリンクを送信

---

### 5.2 ペット管理

#### 基本仕様

- 複数ペット登録可能（上限なし）
- 全画面でアクティブペットを切替可能

#### 複数ペット時の運用ルール

- 表示は **1頭ずつ**（アクティブペット）— 全頭混在表示はしない
- アクティブペット切替ボタンを全主要画面の上部に固定表示する
- アプリ起動時は最後に選択していたペットを表示する
- 通知は**全頭分まとめて**受け取り、本文にペット名を明示する
  - 例：「ポチの投薬時間です（14:00）」「ミケの次回通院は明日です」
- 健康記録・投薬・通院履歴はペット単位で完全に分離して管理する

#### プロフィール項目

| 項目 | 必須 | 内容 |
|------|------|------|
| 名前 | ◎ | テキスト入力 |
| 種類 | ◎ | 犬/猫/うさぎ/鳥/ハムスター/その他 |
| 性別 | 任意 | オス/メス/不明 |
| 誕生日 | 任意 | 日付 |
| 大きさ | 任意 | 小型/中型/大型 |
| 去勢・避妊 | 任意 | 済み/未/不明 |
| 写真 | 任意 | Firebase Storage にアップロード |

#### 仕様

- 後から編集可能
- 論理削除

---

### 5.3 病院管理

#### 基本仕様

- 複数病院を登録・管理可能
- 通院記録から登録済み病院を選択できる

#### 複数病院時の運用ルール

- 病院マスタは**ユーザー単位**で管理する（全ペット共通のリスト）
  - 理由：複数ペットが同じ病院に通うことが多いため、ペットごとに別登録すると重複が発生する
- 通院記録は「どのペット」「どの病院」の組み合わせで保存する
- 病院を削除しても過去の通院記録は消えない
  - 通院記録に病院名（文字列）をコピーして保存するため
- 登録済み病院がない場合・初めての病院の場合は直接入力できる

```
ユーザーの病院リスト（全ペット共通）
├── 〇〇動物病院
└── △△クリニック

通院記録 = ペット（ポチ）× 病院（〇〇動物病院）で保存
通院記録 = ペット（ミケ）× 病院（〇〇動物病院）で保存  ← 同じ病院を別ペットで使える
```

#### 登録項目

| 項目 | 必須 |
|------|------|
| 病院名 | ◎ |
| 電話番号 | 任意 |
| メモ | 任意 |

#### 仕様

- 後から編集可能
- 論理削除（削除しても通院記録の病院名は保持）
- 将来：Google Places API による検索・自動入力

---

### 5.4 ダッシュボード

#### 表示内容（優先順）

1. 未対応の投薬（最優先・赤強調）
2. 次の投薬予定
3. 今日完了済み投薬
4. 薬残量アラート
5. 最新健康記録
6. 次回通院予定
7. 次回トリミング予定

#### 操作

- ワンタップ投薬チェック
- 状態変更
- 健康記録入力

#### アラート条件

- 体重 ±10%以上変化
- 排便異常連続（2回以上）
- 投薬失敗連続
- 水分摂取異常
- 薬残量がX日分以下

---

### 5.5 健康記録

#### 入力フロー

STEP1〜3の順番に表示。STEP2・3は見出しのみ表示し、押下すると展開する。

**STEP1（必須・タップ入力）**

- パッと見の状態（元気・普通・悪い）
- 散歩の有無（行った/行かなかった）
  - 行った場合：散歩時間（時間）

**STEP2（任意・その他の体調）**

- 体重（kg）
- 体温（℃）
- 食事回数
- 食事量（%）
- 水分摂取回数
- 排尿回数
- 排便回数
- 排便タイプ（1〜5）
- 嘔吐回数
- 活動量（1〜4）

**STEP3（任意・病院での診察）**

診察イベントの有無と内容（診察・健康診断・予防接種・薬の受け取り・検査・手術・入院・退院）
※詳細は通院で管理

#### 仕様

- 前回値コピー
- 差分入力
- 未入力OK
- 1日複数登録可能
- 後編集可能
- 論理削除

#### 運用対策

- recordedAt（実入力時間）
- isBackfilled（後入力フラグ）

---

### 5.6 投薬管理

#### 薬登録

| 項目 | 必須 |
|------|------|
| 薬名 | ◎ |
| 用量 | ◎ |
| 投薬回数（1日） | ◎ |
| 投薬時間 | ◎ |
| 開始日 | ◎ |
| 終了日 | 任意 |
| メモ | 任意 |

#### 残量管理

- 現在の残量（錠数 or ml）
- 1回あたりの使用量
- 残日数の自動計算（残量 ÷ 1日使用量）
- 残量アラート（残X日分以下で通知）

#### 投薬スケジュール生成

処方保存時に自動生成。例：1日2回 × 14日 → 28件

#### 状態

| 状態 | 説明 |
|------|------|
| scheduled | 未来の予定 |
| pending | 対応待ち（当日・時刻到達） |
| done | 投薬完了 |
| skipped | 意図的にスキップ |
| failed | 投薬失敗 |
| delayed | 予定時刻を過ぎて実施 |

#### 操作

- ワンタップ完了
- 状態変更
- まとめてチェック

#### 仕様

- 複数時間対応
- 期間指定
- 後編集可能
- 当日一括入力可能
- 遅延を正式状態として扱う
- 今日のみスケジュール変更可能

#### 再生成ルール

処方変更時：future（scheduled/pending）のみ再生成、done 履歴は保持

| 用量 | ◎ |

| 日数 | ◎ |

#### 仕様

- 変更時は未実施スケジュールのみ再生成

### 5.10 トリミング管理

#### 設計方針

スケジュール（予定）と記録（実績）を**1つのエントリーで一体管理**する。

```
予約時：予定日・サロン名だけ入力して保存
終了後：実施日・費用・内容を追記して更新

状態
├── 予定（completedDate = null）
└── 完了（completedDate に日付が入っている）
```

#### 入力項目

| 項目 | タイミング | 必須 |
|------|-----------|------|
| 予定日 | 予約時 | ◎ |
| サロン名 | 予約時 | 任意（テキスト入力） |
| メモ | 予約時 | 任意 |
| 実施日 | 終了後 | ◎（完了時） |
| 施術内容 | 終了後 | 任意 |
| 費用 | 終了後 | 任意 |
| 次回予定日 | 終了後 | 任意 |

#### 画面上の表示

- 一覧：予定・完了を分けて表示
- 予定のものは通知対象（前日・当日）

#### サロン管理

- サロンマスタは**ユーザー単位**で管理（病院と同じ設計）
- トリミング登録時に登録済みサロンから選択、または直接入力
- サロンを削除してもトリミング記録のサロン名は保持（名前をコピーして保存）

#### 仕様

- 予約時に入力して、終了後に同じエントリーを追記・更新する
- 後から編集可能
- 論理削除

---

### 5.11 通知

#### 種類・優先度・タイミング

| 通知 | 優先度 | タイミング |
|------|--------|-----------|
| 投薬リマインド | 高 | 予定時間の5分前 |
| 未対応投薬リマインド | 高 | 予定時間から1時間後 |
| 薬残量アラート | 高 | 残X日分を下回ったとき |
| 次回通院リマインド | 中 | 前日・当日 |
| 次回トリミングリマインド | 中 | 前日・当日 |
| 健康記録入力促し | 低 | 1日1回（未入力時） |

#### 複数ペット時の通知ルール

- 通知本文には必ずペット名を含める（例：「ポチの投薬時間です」）
- 複数ペットの投薬が同時刻の場合は1件にまとめず個別に通知する
- 通知のON/OFFはペット単位ではなく通知種別単位で設定する

#### 制御

- 対応済みは通知しない
- 重複防止
- 通知ごとにON/OFF設定可能

---

### 5.12 設定

- 健康記録リマインド ON/OFF
- 各種通知 ON/OFF
- ペット管理
- 病院管理
- アカウント管理

## 7. データ管理単位

| データ | 管理単位 | 理由 |
|--------|----------|------|
| 健康記録 | ペット単位 | 各ペットの体調は独立 |
| 投薬・スケジュール | ペット単位 | 薬はペットごとに異なる |
| 通院記録 | ペット単位 | どのペットが受診したかを管理 |
| 処方 | ペット単位 | 処方はペットに紐づく |
| トリミング | ペット単位 | トリミングはペットごとに管理 |
| 病院マスタ | ユーザー単位 | 複数ペットで同じ病院を共有できる |
| サロンマスタ | ユーザー単位 | 複数ペットで同じサロンを共有できる |

## 9. 状態遷移

### 投薬ログ

```
scheduled → pending（当日・時刻到達）
pending   → done（投薬完了）
pending   → skipped（スキップ）
pending   → failed（失敗）
pending   → delayed（遅延）
delayed   → done（遅れて実施）
```

## 10. 非機能要件

- ダッシュボード表示：1秒以内

- 対応デバイス：モバイル前提（最大幅480px）

## 11. 運用課題と対策

| 課題 | 対策 |
|------|------|
| 通知疲れ | 優先度制御・対応済み通知を非表示 |
| データ不正確 | 後入力フラグ・入力時間保存 |
| 投薬スケジュール崩壊 | 遅延状態を正式対応・柔軟時間設計 |
| 多頭飼い混乱 | 全画面でアクティブペットを明示 |
| データ肥大化 | ページネーション・最新優先取得 |
