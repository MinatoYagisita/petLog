# 受入テスト支援 Skill

> 前提: Playwright（E2E テスト）/ Vercel Preview 環境での確認

## この Skill を使う場面

- お客様受入テストの観点を事前に整理するとき
- 受入テストで出た指摘を整理・分類するとき
- 指摘への対応方針を検討するとき

## 入力

- 納品対象の機能一覧
- 仕様書（受入テストの基準となるもの）
- お客様からの指摘票・フィードバック（ある場合）

## 実施手順

### Step 1: 受入テスト観点の整理

お客様の立場で確認すべき観点を洗い出す。

| カテゴリ | 確認観点 |
|----------|---------|
| 仕様一致 | 仕様書通りに動作するか |
| 業務フロー | 実際の業務手順に沿って操作できるか |
| 画面表示 | ラベル・レイアウト・日本語が正しいか |
| データ整合性 | 登録・更新・削除後のデータが正しいか |
| エラー処理 | 不正操作時に適切なメッセージが出るか |
| パフォーマンス | 実用上問題ない応答速度か |
| 帳票・出力 | 帳票やCSV出力が仕様通りか |

### Step 2: 指摘票の整理

お客様からの指摘を以下のフォーマットで整理する。

```markdown
| 指摘ID | 指摘内容 | 対象画面 | 分類 | 優先度 | ステータス | 対応方針 |
|--------|---------|---------|------|--------|-----------|---------|
| FB-001 | 一覧のソート順が逆 | ユーザー一覧 | 不具合 | P1 | 未対応 | 修正対応 |
| FB-002 | 検索に部分一致が欲しい | ユーザー検索 | 追加要望 | P3 | 協議中 | 次期対応を提案 |
```

### Step 3: 優先度の判定（P1 / P2 / P3）

| 優先度 | 基準 | 対応目安 |
|--------|------|---------|
| P1（高） | 業務が止まる、データが壊れる、仕様と明確に異なる | 即時対応 |
| P2（中） | 業務は回避策があるが不便、表示崩れ、軽微な仕様差異 | 納品前に対応 |
| P3（低） | あると便利、見た目の好み、仕様範囲外の改善要望 | 次期検討 or 対応不要 |

#### 判定のポイント

- **業務影響**: その不具合でお客様の業務が止まるか
- **回避策**: ワークアラウンドがあるか
- **仕様明記**: 仕様書に明記された動作と異なるか
- **頻度**: 毎回発生するか、特定条件のみか

### Step 4: 仕様内 / 仕様外の切り分け

指摘が仕様範囲内か外かを判定する。

| 分類 | 定義 | 対応方針 |
|------|------|---------|
| 不具合（仕様内） | 仕様書に書かれた動作と実際が異なる | 修正対応。開発側の責任 |
| 仕様漏れ | 仕様書に記載がないが合理的に期待される動作 | 協議の上で対応方針を決定 |
| 追加要望（仕様外） | 仕様書に記載がなく、要望として新規に出たもの | 次期対応 or 別見積もり |

#### 切り分けの考え方

- 仕様書の該当箇所を引用して判断根拠を示す
- 「仕様書に書いていない = 仕様外」と機械的に判断しない
- 業界標準やユーザビリティの観点で合理的に期待される動作は「仕様漏れ」として協議する
- 判断に迷う場合は PM に相談する

### Step 5: 軽微修正と追加要望の切り分け

| 分類 | 例 | 対応 |
|------|----|------|
| 軽微修正 | 誤字修正、ラベル変更、余白調整 | 現行スコープ内で対応 |
| 追加要望 | 新機能追加、大幅なUI変更、新規帳票 | スコープ外として別途見積もり |
| グレーゾーン | 仕様書に明記がないが小規模な改善 | PM判断で対応可否を決定 |

## 出力フォーマット

### 受入テスト結果レポート

```markdown
## 受入テスト結果レポート

### 概要
- 対象機能: ユーザー管理機能
- テスト期間: 2024-01-20 〜 2024-01-22
- 指摘件数: 8件

### 指摘サマリ

| 分類 | P1 | P2 | P3 | 合計 |
|------|----|----|----|----|
| 不具合 | 1 | 2 | 0 | 3 |
| 仕様漏れ | 0 | 1 | 0 | 1 |
| 追加要望 | 0 | 0 | 3 | 3 |
| 軽微修正 | 0 | 0 | 1 | 1 |

### 対応方針
- P1 不具合: 即時修正し再テスト依頼
- P2 不具合・仕様漏れ: 納品前に対応
- P3 追加要望: 次期開発で検討を提案
- 軽微修正: 現行スコープ内で対応
```

## チェック観点

- [ ] 全指摘に優先度と分類を設定したか
- [ ] 仕様内 / 仕様外の根拠を示したか
- [ ] P1 の指摘がすべて対応済み or 対応中か
- [ ] 追加要望を不具合と混同していないか
- [ ] レポートが PM・お客様に提出可能な形式か

## 禁止事項

- 指摘を受けたら全て即座に対応しようとしない（分類・優先度付けを先にする）
- 仕様外の要望を「軽微だから」と無断でスコープに入れない
- お客様への回答を PM の確認なく返さない
- 判断に迷う指摘を自分だけで判断しない
