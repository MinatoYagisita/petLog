# 境界値・バリデーションテスト Skill

> 前提: TypeScript / Next.js / Zod(バリデーション) / Vitest / Playwright

## この Skill を使う場面

- フォーム入力のバリデーションテストを設計するとき
- 境界値テストの抜け漏れを確認したいとき
- 入力項目が多い画面のテスト観点を整理するとき

## 入力

- 画面仕様書（入力項目の一覧、各項目の制約）
- API 仕様書（パラメータの型・制約）
- 既存のバリデーションルール

## 実施手順

### Step 1: 入力項目の整理

対象画面・API の全入力項目を一覧化し、以下を整理する。

| 項目名 | 型 | 必須 | 最小 | 最大 | 形式 | その他制約 |
|--------|-----|------|------|------|------|-----------|
| メール | string | Yes | - | 255文字 | email形式 | 重複不可 |
| パスワード | string | Yes | 8文字 | 64文字 | 英数記号 | - |
| 年齢 | integer | No | 0 | 150 | - | - |
| 生年月日 | date | No | 1900-01-01 | 本日 | YYYY-MM-DD | - |

### Step 2: 典型パターンの適用

各項目に対して、以下の典型パターンを適用する。

#### 文字列項目

| パターン | テスト値 | 期待 |
|----------|---------|------|
| 未入力（必須） | 空文字 / null | エラー |
| 未入力（任意） | 空文字 / null | OK |
| 最小文字数 | ちょうど最小 | OK |
| 最小文字数 - 1 | 最小 - 1文字 | エラー |
| 最大文字数 | ちょうど最大 | OK |
| 最大文字数 + 1 | 最大 + 1文字 | エラー |
| 半角スペースのみ | " " | 要確認 |
| 前後にスペース | " abc " | トリム有無確認 |
| HTML タグ | `<script>alert(1)</script>` | エスケープ / エラー |
| 制御文字 | タブ、改行 | 要確認 |
| マルチバイト | 日本語、絵文字 | 文字数カウント確認 |

#### 数値項目

| パターン | テスト値 | 期待 |
|----------|---------|------|
| 未入力（必須） | 空 / null | エラー |
| 最小値 | ちょうど最小 | OK |
| 最小値 - 1 | 最小 - 1 | エラー |
| 最大値 | ちょうど最大 | OK |
| 最大値 + 1 | 最大 + 1 | エラー |
| 0 | 0 | 仕様確認 |
| 負数 | -1 | 仕様確認 |
| 小数 | 1.5 | 整数制約ならエラー |
| 文字列 | "abc" | エラー |
| 非常に大きな値 | 999999999999 | エラー or 上限 |

#### 日付項目

| パターン | テスト値 | 期待 |
|----------|---------|------|
| 未入力（必須） | 空 / null | エラー |
| 下限日 | ちょうど下限 | OK |
| 下限日 - 1日 | 下限の前日 | エラー |
| 上限日 | ちょうど上限 | OK |
| 上限日 + 1日 | 上限の翌日 | エラー |
| 不正形式 | "2024/13/40" | エラー |
| 閏年 | 2024-02-29 | OK |
| 閏年でない年 | 2023-02-29 | エラー |
| 存在しない日 | 2024-04-31 | エラー |

#### 選択項目（セレクト・ラジオ・チェックボックス）

| パターン | テスト値 | 期待 |
|----------|---------|------|
| 未選択（必須） | 選択なし | エラー |
| 有効な値 | 選択肢内の値 | OK |
| 不正な値（API直接） | 選択肢外の値 | エラー |

### Step 3: TypeScript / Next.js フォーム固有の観点

| パターン | テスト値 | 期待 | 確認場所 |
|----------|---------|------|---------|
| クライアントバリデーション通過 + サーバー拒否 | クライアント JS を迂回して直接 API 送信 | サーバーでエラー | Vitest (API Route) |
| Zod スキーマの型不一致 | `{ age: "abc" }` を number フィールドに | parse エラー | Vitest |
| FormData の不正キー | 存在しないフィールド名を送信 | 無視 or エラー | Vitest |
| Server Action の直接呼び出し | 不正な引数で Server Action を呼ぶ | サーバーでエラー | Vitest |
| `undefined` / `null` の区別 | 未送信 vs 空文字送信 | 仕様に応じた挙動 | Vitest |

### Step 4: 最低限ラインの確認

以下は **どんな入力項目でも最低限テストすること**。

- [ ] 必須項目の未入力
- [ ] 最大文字数 / 最大値の境界
- [ ] 型が違う入力（数値項目に文字列など）
- [ ] XSS を意図した入力（`<script>` タグ等）
- [ ] 重複がありうる項目の重複登録
- [ ] **クライアントバリデーション「だけ」でなく、サーバーサイドでも拒否されること**

### Step 5: テストケース化

洗い出した観点をテストケース表にまとめる（test-spec-creator の出力形式に合わせる）。
各ケースに **Vitest / Playwright の振り分け** を明記する。

## 出力フォーマット

```markdown
## 境界値・バリデーションテストケース: [画面名 / API名]

### 項目別テストケース

#### メールアドレス
| TC-ID | パターン | テスト値 | 期待結果 | 優先度 |
|-------|---------|---------|---------|--------|
| BV-001 | 未入力 | 空文字 | エラー: メールアドレスは必須です | 高 |
| BV-002 | 正常値 | test@example.com | OK | 高 |
| BV-003 | 形式不正 | test@example | エラー: メール形式不正 | 高 |
| BV-004 | 最大長 | a{246}@example.com (255文字) | OK | 中 |
| BV-005 | 最大長+1 | a{247}@example.com (256文字) | エラー | 中 |
| BV-006 | 重複 | 登録済みアドレス | エラー: 登録済み | 高 |
| BV-007 | XSS | <script>alert(1)</script>@a.com | エラー | 高 |
```

## チェック観点

- [ ] 全入力項目を洗い出したか
- [ ] 必須 / 任意の区別は正しいか
- [ ] 境界値（ちょうど / ±1）を両方テストしているか
- [ ] 型不正のテストを含めているか
- [ ] XSS 等のセキュリティ観点を含めているか
- [ ] 最低限ラインをすべて満たしているか

## 禁止事項

- 「全項目に正常値を入れて通った」だけで完了としない
- 仕様書に制約が書かれていない項目を「制約なし」と決めつけない（確認事項にする）
- クライアントサイドのバリデーションだけ確認してサーバーサイドを省略しない
- 境界値の片側（上限だけ、下限だけ）しかテストしない状態で完了としない
