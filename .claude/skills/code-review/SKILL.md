# コードレビュー Skill

> 前提: TypeScript / Next.js / Prisma / Auth.js / Vitest / Playwright
> **このSkillの出力は PM レビューの補助であり、最終承認は人間が行う。**

## この Skill を使う場面

- PR（プルリクエスト）のレビューを行うとき
- PM レビュー前に自己レビューするとき
- 他メンバーのコードをレビューするとき

## 入力

- レビュー対象のコード（差分 or ファイル）
- 対応する仕様書・チケット（あれば）
- 既存のコーディングルール（`.claude/rules/code-style.md` を参照）

## 実施手順

### Step 1: 仕様一致の確認

- [ ] 実装が仕様・チケットの要件を満たしているか
- [ ] 仕様にある条件分岐が漏れなく実装されているか
- [ ] 仕様にない機能を勝手に追加していないか
- [ ] 仕様の曖昧な部分を勝手に解釈して実装していないか

### Step 2: 命名・可読性・型安全性の確認

- [ ] 変数名・関数名が処理内容を適切に表しているか（camelCase / PascalCase ルール）
- [ ] 略語や暗号的な名前を使っていないか
- [ ] 関数の責務が単一か（1つの関数が複数の仕事をしていないか）
- [ ] ネストが深すぎないか（3段以上のネストがないか）
- [ ] 不要なコメントアウトが残っていないか
- [ ] `console.log` やデバッグ用コードが残っていないか
- [ ] `any` 型を使っていないか（使う場合はコメントで理由が書かれているか）
- [ ] `as` キャストを安易に使っていないか（型ガードを優先しているか）

### Step 3: 例外処理・エラーハンドリングの確認

- [ ] 例外を握りつぶしていないか（`catch` で何もしないなど）
- [ ] エラーメッセージがユーザーにとって分かりやすいか
- [ ] 異常系で適切なステータスコードを返しているか
- [ ] 外部 API 呼び出しのエラーハンドリングがあるか

### Step 4: セキュリティの確認

- [ ] ユーザー入力をサーバーサイドでバリデーションしているか（Zod スキーマ等）
- [ ] Prisma の `$queryRaw` でパラメータバインドを使っているか
- [ ] `dangerouslySetInnerHTML` を使っていないか（XSS）
- [ ] Auth.js の `getServerSession()` で認証チェックしているか
- [ ] 秘密情報がハードコードされていないか
- [ ] `.env` の値がクライアントに露出していないか（`NEXT_PUBLIC_` の使い方）

### Step 5: テストの確認

- [ ] 新規・変更した機能にテストが書かれているか
- [ ] ロジックは Vitest、画面操作は Playwright で適切に振り分けられているか
- [ ] テストが正常系だけでなく異常系も含んでいるか
- [ ] テストが仕様に基づいているか（実装に依存したテストでないか）
- [ ] テストカバレッジが妥当か

### Step 6: 保守性の確認

- [ ] 同じ処理のコピペがないか（共通化すべき箇所）
- [ ] マジックナンバーや固定文字列が定数化されているか
- [ ] 将来の変更に対して影響範囲が局所化されているか
- [ ] 不要になったコードが削除されているか

### Step 7: パフォーマンスの確認（初歩的な観点）

- [ ] Prisma で N+1 クエリが発生していないか（`include` でまとめているか）
- [ ] `prisma.xxx.findMany()` に `take` / `skip` が付いているか
- [ ] 不要なループ内での DB 問い合わせがないか
- [ ] 大量データの全件取得をしていないか
- [ ] 不要な再レンダリングが発生しないか（`useMemo` / `useCallback` の検討）
- [ ] `"use client"` が不要なコンポーネントに付いていないか

### Step 8: Next.js / Prisma 固有の確認

- [ ] Server Components と Client Components が適切に分離されているか
- [ ] クライアントから直接 Prisma を呼んでいないか
- [ ] API Route / Server Actions でセッション確認を行っているか
- [ ] `select` / `include` で必要なフィールドだけ取得しているか
- [ ] マイグレーションファイルを手動で編集していないか

## 出力フォーマット

レビュー結果はコメント一覧として出力する。

```markdown
## コードレビュー結果

- **対象**: PR #123 / [チケット名]
- **レビュー日**: 2024-01-20
- **レビュアー**: [名前]

### レビューコメント一覧

| # | ファイル | 行 | カテゴリ | 重要度 | 指摘内容 | 提案 |
|---|---------|-----|---------|--------|---------|------|
| 1 | app/api/users/route.ts | 45 | セキュリティ | 高 | `$queryRawUnsafe` で文字列結合している | `Prisma.sql` テンプレートに変更 |
| 2 | app/api/orders/route.ts | 12 | 例外処理 | 中 | catch ブロックが空 | ログ出力 + 適切なステータスコードを返す |
| 3 | components/UserList.tsx | 78 | パフォーマンス | 中 | useEffect 内で全件取得している | Server Component でページネーション追加 |
| 4 | lib/utils.ts | 5 | 命名 | 低 | 関数名 `proc` が曖昧 | `processUserRegistration` に変更 |

### サマリ

| 重要度 | 件数 |
|--------|------|
| 高（修正必須） | 1 |
| 中（修正推奨） | 2 |
| 低（改善提案） | 1 |

### 総評
- セキュリティの高重要度指摘が1件あり、マージ前に修正が必要
- 全体的に可読性は良好
```

## チェック観点

- [ ] 全8ステップの観点を確認したか
- [ ] 指摘に重要度を設定したか
- [ ] 指摘に対する具体的な改善提案を含めたか
- [ ] 高重要度の指摘がある場合、マージ可否の判断を示したか

## 禁止事項

- 重要度「高」の指摘を無視してマージ可の判断をしない
- コードスタイルの好みだけで指摘しない（ルールに基づく）
- 「全体的に良いと思います」だけのレビューをしない
- 人間の PM レビューを代替するものとして扱わない（あくまで補助）
- レビュー対象のコードを読まずに一般論だけで指摘しない
