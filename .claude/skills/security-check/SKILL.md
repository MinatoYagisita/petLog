# セキュリティチェック Skill

> 前提: Next.js / Auth.js / Prisma / PostgreSQL / Vercel / pnpm

## この Skill を使う場面

- 納品前のセキュリティ確認を行うとき
- 新しい外部 Skills やライブラリの採用を審査するとき
- コードレビューでセキュリティ観点を確認するとき

## 入力

- 対象のソースコード
- 使用しているライブラリ・依存関係の一覧
- （外部 Skills 審査の場合）対象 Skills の情報

## 実施手順

### Step 1: アプリケーションセキュリティチェック

以下のチェックリストを順に確認する。

#### 入力値検証

- [ ] すべてのユーザー入力にサーバーサイドバリデーションがあるか
- [ ] クライアントサイドのバリデーションだけに依存していないか
- [ ] 入力長の上限が設定されているか
- [ ] 型チェックが行われているか

#### SQL インジェクション対策（Prisma）

- [ ] `prisma.$queryRaw` で `Prisma.sql` テンプレートリテラルを使っているか
- [ ] `prisma.$executeRawUnsafe` を使っていないか（使う場合は PM 承認済みか）
- [ ] 文字列結合で SQL を組み立てている箇所がないか
- [ ] Prisma の通常メソッドでは問題ないが、raw クエリの箇所を重点確認したか

#### XSS 対策（React / Next.js）

- [ ] `dangerouslySetInnerHTML` を使っていないか
- [ ] URL パラメータを直接 DOM に反映していないか
- [ ] React の JSX デフォルトエスケープに頼れない箇所がないか

#### CSRF 対策

- [ ] Auth.js の Cookie ベースセッション保護が有効か
- [ ] Bearer トークンのみの API の場合、CSRF 不要の理由を記録しているか

#### 認証・認可（Auth.js）

- [ ] API Route / Server Actions で `getServerSession()` による認証チェックがあるか
- [ ] 権限チェックがサーバーサイドで行われているか（クライアントの非表示に依存していないか）
- [ ] Auth.js のプロバイダ設定が適切か
- [ ] セッションの有効期限が適切か

#### 情報漏洩防止

- [ ] エラーレスポンスにスタックトレースや内部情報が含まれていないか
- [ ] ログに機密情報（パスワード、トークン等）が記録されていないか
- [ ] ソースコードにハードコードされた秘密情報がないか
- [ ] `.env` / `.env.local` / `.env.production` が `.gitignore` に含まれているか
- [ ] `NEXT_PUBLIC_` プレフィックスなしの環境変数がクライアントに露出していないか

#### ファイルアップロード

- [ ] アップロード可能な拡張子を制限しているか
- [ ] MIME タイプの検証をサーバーサイドで行っているか
- [ ] ファイルサイズの上限を設定しているか

#### 初期アカウント・初期パスワード

- [ ] 初期アカウントにデフォルトパスワードが設定されたまま残っていないか
- [ ] 初回ログイン時にパスワード変更を強制しているか

#### CSRF 該当性の確認

- [ ] CSRF 対策が不要な構成（Bearer トークン認証のみ等）の場合、その理由を記録しているか

### Step 2: 依存関係・サプライチェーンの確認

#### 既知の脆弱性

- [ ] `pnpm audit` で既知の脆弱性がないか確認
- [ ] 使用しているライブラリのバージョンが古すぎないか
- [ ] メンテナンスが停止しているライブラリがないか
- [ ] 不要なライブラリが含まれていないか

#### サプライチェーン攻撃対策

- [ ] `pnpm install --frozen-lockfile` が CI / 本番デプロイで使われているか
- [ ] `package.json` のバージョンが `^` ではなく固定されているか（`pnpm add --save-exact`）
- [ ] 新規追加パッケージに `postinstall` / `preinstall` / `prepare` スクリプトがないか確認したか
- [ ] `pnpm.onlyBuiltDependencies` でビルドスクリプト許可パッケージを明示しているか
- [ ] 主要な依存パッケージについて `pnpm why パッケージ名` で推移的依存を確認したか
- [ ] lockfile の変更差分をレビューしたか

> **注意**: `pnpm audit` は既知脆弱性の検出を補助するツールであり、安全性を保証するものではない。メンテナーアカウント乗っ取りによる正規パッケージへのマルウェア混入（サプライチェーン攻撃）は `audit` では検出できない。外部通信、情報取得、実行内容、`postinstall` 等はツール任せにせず、別途コードを目視確認すること。

### Step 3: AI 利用時の機密保持確認

- [ ] **AI ツールに機密情報を入力していないことを確認する**（「残っていないか確認する」ではなく「そもそも入れない」が基本）
- [ ] やむを得ず機密情報を扱う場合は、PM承認済みの方法・環境のみで扱っているか
- [ ] AI が生成したコードに秘密情報がハードコードされていないか
- [ ] `.claude` 配下に機密情報を含むファイルがないか

### Step 4: 外部Git由来資産の情報漏洩チェック（該当時のみ）

外部Gitリポジトリの Skills / rules / スクリプト等を採用候補として評価する場合、**情報漏洩リスクを最優先で確認する**。

#### 4-1. 外部通信の有無

- [ ] HTTP / HTTPS リクエストを送信する処理がないか
- [ ] Webhook / analytics / telemetry / metrics 送信がないか
- [ ] 外部 API へのアクセスがないか
- [ ] 明示されていない通信処理がないか
- [ ] 実行時に外部ホストへ勝手に接続しないか

#### 4-2. 情報抜き取りの可能性

- [ ] 環境変数を読み取って送信していないか
- [ ] APIキー、トークン、認証情報を収集していないか
- [ ] ローカルファイルを読み取って送信していないか
- [ ] Git設定、SSH鍵、`.env`、認証ファイル、履歴ファイルを参照していないか
- [ ] クリップボード、ホームディレクトリ、設定ディレクトリを不必要に読んでいないか
- [ ] OS情報、ユーザー名、ホスト名、ネットワーク設定等を不必要に収集していないか
- [ ] shell history、git config、ブラウザ設定、ローカル環境情報を広く収集していないか

#### 4-3. 実行コードの危険性

- [ ] `curl | bash` 的な実行をしていないか
- [ ] `npx` 実行で外部コードをそのまま動かさないか
- [ ] `postinstall` / `prepare` / `preinstall` / `install` スクリプトが危険でないか
- [ ] シェルスクリプト内で不明なダウンロードや送信処理がないか
- [ ] 難読化されたコード、圧縮コード、読みにくいワンライナーがないか
- [ ] 実行時に追加ファイルや追加コードをダウンロードしていないか
- [ ] 自己更新機構やリモートスクリプト取得の仕組みがないか

#### 4-4. 権限・スコープの過剰さ

- [ ] 本来不要なファイルアクセス権を要求していないか
- [ ] 全ディレクトリ走査やホーム配下の再帰探索をしていないか
- [ ] Gitリポジトリ全体を不要に収集していないか
- [ ] 「skill」と関係ない広い権限や処理を持っていないか

#### 4-5. 依存関係の安全性

- [ ] 不審な依存パッケージがないか
- [ ] 依存先に postinstall 等の危険な処理がないか
- [ ] 最近作られた怪しいパッケージや名前が紛らわしいパッケージでないか
- [ ] skill 本体より依存関係側に危険が潜んでいないか

#### 4-6. 基本情報の確認

| 確認項目 | 確認内容 | 結果 |
|----------|---------|------|
| 提供元 | 信頼できる組織 / 個人か | |
| ソースコード公開 | コードが公開され、内容を確認できるか | |
| 更新頻度 | 直近6ヶ月以内に更新されているか | |
| 利用実績 | 他プロジェクトでの採用実績 | |
| ライセンス | ライセンスが商用利用に適合するか | |
| README | ドキュメントの記載内容が妥当か | |

### Step 5: npx 利用時の確認

- [ ] パッケージ名が正しいか（typosquatting でないか）
- [ ] パッケージの公開元を確認したか
- [ ] パッケージの weekly downloads と最終更新日を確認したか
- [ ] 実行内容を把握しているか
- [ ] `postinstall` / `preinstall` / `prepare` スクリプトの中身を確認したか
- [ ] PM の承認を得ているか

## 外部Git由来資産の判定基準

| 判定 | 基準 |
|------|------|
| **採用可** | 重大な外部通信・情報取得・危険な実行処理がなく、依存関係も把握できている |
| **条件付き可** | 一部未確認点があるが、限定導入・自社向け再構成・追加検証を条件に許容できる |
| **不採用** | 業務上不要な外部送信、情報取得の過剰さ、危険なスクリプト、難読化、説明不能な挙動のいずれかがある |

- 「条件付き可」の場合は、条件と期限を明記し、条件が満たされるまで本番利用しない
- 判断に迷う場合は「不採用」に倒す

## チェック観点

- [ ] 全カテゴリのチェックを実施したか
- [ ] NG 項目に対応方針を記載したか
- [ ] 重要度の判定は妥当か
- [ ] 外部依存のリスク評価を行ったか
- [ ] 外部Git由来資産の場合、情報漏洩チェック（Step 4）を実施したか
- [ ] 外部Git由来資産の場合、総合判定と判定理由を記載したか

## 禁止事項

- チェックリストを形式的に全部 OK にしない
- 「フレームワークが守ってくれるから大丈夫」と確認を省略しない
- セキュリティの NG 項目を「軽微」として放置しない
- 外部 Skills の採用を PM 承認なく決定しない
- 実際のコードを確認せず、推測だけで OK 判定しない
- 情報漏洩リスクが疑われる外部資産を「便利だから」で採用しない
- 難読化されたコードを含む外部資産を採用しない
- 判断に迷う外部資産を「とりあえず採用」しない（迷ったら不採用）
