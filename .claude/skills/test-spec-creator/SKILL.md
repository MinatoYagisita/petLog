# テスト仕様書作成 Skill

> 前提: Vitest（単体・結合）/ Playwright（E2E・受入）

## この Skill を使う場面

- 新規機能の仕様書をもとにテストケースを作成するとき
- 既存機能の改修時にテスト観点を洗い出すとき
- テストケースの抜け漏れを確認したいとき

## 入力

- 仕様書（機能仕様、画面仕様、API 仕様など）
- 対象機能の範囲
- 既存テストケースがあればそれも提供する

## 実施手順

### Step 1: 仕様の整理

1. 仕様書を読み、対象機能の入力・処理・出力を整理する
2. 画面がある場合は画面項目と操作を一覧化する
3. 関連する他機能への影響を洗い出す

### Step 2: テスト観点の洗い出し

以下の6観点を必ず検討する。

| # | 観点 | 確認内容 |
|---|------|----------|
| 1 | 正常系 | 仕様通りの入力で期待通り動作するか |
| 2 | 異常系 | 不正入力・エラー条件で適切にハンドリングされるか |
| 3 | 境界値 | 上限・下限・ちょうどの値で正しく動くか |
| 4 | 権限 | 操作権限による制御が正しいか |
| 5 | 重複 | 重複データの処理が正しいか |
| 6 | 画面遷移 | 操作後の遷移先・状態が正しいか |

### Step 3: Vitest / Playwright の振り分け

各テストケースを以下の基準でツールに振り分ける。

| テスト対象 | ツール | 理由 |
|-----------|--------|------|
| 関数・ユーティリティのロジック | Vitest | 高速に回せる |
| API Route のリクエスト/レスポンス | Vitest | サーバーサイドのテスト |
| Prisma を使った DB 操作 | Vitest（テスト用DB接続） | 結合テスト |
| React コンポーネントの表示 | Vitest + Testing Library | DOM テスト |
| 画面操作・ブラウザ上の動作 | Playwright | E2E |
| 業務フロー全体・画面遷移 | Playwright | ユーザー体験の確認 |
| 権限による画面制御 | Playwright | 実ブラウザでの認証フロー |

### Step 4: テストケースの作成

各観点について具体的なテストケースを作成し、ツール振り分けを明記する。

### Step 5: 確認事項の切り出し

仕様に記載がない、または曖昧な箇所は「確認事項」として別途リストアップする。

## 出力フォーマット

### テストケース表

```markdown
| TC-ID | 観点 | ツール | テスト項目 | 前提条件 | 操作手順 | 期待結果 | 優先度 |
|-------|------|--------|-----------|---------|---------|---------|--------|
| TC-001 | 正常系 | Playwright | 有効な情報で登録できる | 未登録状態 | 1. 必須項目を全て入力 2. 登録ボタン押下 | 登録成功メッセージが表示される | 高 |
| TC-002 | 異常系 | Vitest | 必須項目未入力で API エラー | - | POST /api/users に必須項目なしで送信 | 400 エラーが返る | 高 |
| TC-003 | 境界値 | Vitest | 文字数上限ちょうどで登録できる | - | validateName("a".repeat(50)) | true が返る | 中 |
```

### 確認事項リスト

```markdown
## 確認事項
- [ ] Q1: 名前の文字数上限は何文字か（仕様書に記載なし）
- [ ] Q2: 重複チェックの対象はメールアドレスのみか
- [ ] Q3: 削除後に同じメールアドレスで再登録は可能か
```

## チェック観点

- [ ] 6観点（正常系/異常系/境界値/権限/重複/画面遷移）すべてを検討したか
- [ ] 各テストケースに前提条件と期待結果が明記されているか
- [ ] 曖昧な仕様を勝手に解釈していないか
- [ ] 優先度が設定されているか
- [ ] 確認事項が漏れなく切り出されているか

## 禁止事項

- 仕様に書かれていない動作を「こうだろう」と推測してテストケースにしない
- テスト観点の洗い出しを省略して、いきなりテストコードを書かない
- 正常系だけ作って完了としない
- 出力を PM に確認せず確定しない

## 例

### 入力例

> ユーザー登録機能の仕様書を渡すので、テストケースを作成してください。
> - メールアドレスとパスワードで登録
> - メールアドレスは重複不可
> - パスワードは8文字以上

### 出力例（抜粋）

| TC-ID | 観点 | ツール | テスト項目 | 前提条件 | 操作手順 | 期待結果 | 優先度 |
|-------|------|--------|-----------|---------|---------|---------|--------|
| TC-001 | 正常系 | Playwright | 有効なメール・パスワードで登録 | 未登録 | 1. /register で入力 2. 登録押下 | 登録成功画面に遷移 | 高 |
| TC-002 | 異常系 | Vitest | メール未入力で API エラー | - | POST /api/auth/register に email なしで送信 | 400 エラー | 高 |
| TC-003 | 異常系 | Vitest | 不正なメール形式 | - | validateEmail("abc") | false | 高 |
| TC-004 | 境界値 | Vitest | パスワード7文字 | - | validatePassword("Abc1234") | false | 高 |
| TC-005 | 境界値 | Vitest | パスワード8文字 | - | validatePassword("Abc12345") | true | 高 |
| TC-006 | 重複 | Vitest | 登録済みメールで再登録 | DB に test@example.com 存在 | createUser({email: "test@example.com"}) | 重複エラー | 高 |

#### 確認事項
- [ ] Q1: パスワードの文字数上限はあるか
- [ ] Q2: パスワードに使用可能な文字種の制限はあるか
- [ ] Q3: メールアドレスの確認メール送信はあるか
